Title: Hash-First, Range-Second RPC Ingestion: A Data-Structure Design for web3.py Under Ethereum Throughput Limits

Thesis:
The right way to build high-throughput Ethereum readers in Python is to treat web3.py as a transport layer, Ethereum RPC performance limits as the hard execution budget, and data structures as the core control plane: use hash-based dict/set indexes for O(1)-average deduplication and membership checks, then layer sorted/range-oriented structures only for scan and replay windows, because provider throughput windows, batch constraints, and historical-range caps make naive sequential RPC fetching structurally inefficient [Python Wiki: TimeComplexity; CPython setobject.c documentation references via Python docs][web3.py transactions documentation v7.1.0][Alchemy Docs: Throughput][Alchemy Docs: Batch Requests][QuickNode guide: How To Make Batch Requests on Ethereum; QuickNode Ethereum docs][Chainstack Throughput Guidelines].

The System:
Build one ingestion and query system with three coordinated layers.

1. Ingest with the correct web3.py provider mode.
Use AsyncWeb3 with AsyncHTTPProvider or AsyncIPCProvider for async production workflows, and use WebSocketProvider or AsyncIPCProvider when subscription support is required, because subscriptions require a persistent provider [web3.py providers documentation v7.15.0/v7.0.0][web3.py Event Subscriptions documentation v7.15.0]. When the Python process runs on the same machine as the node, prefer IPC because the provider docs recommend IPC for best performance [web3.py providers documentation v7.15.0/v7.0.0].

2. Make the primary live index a hash index.
Use Python dict/set as the first data structure for every event, transaction hash, log key, or block marker that needs deduplication or “already seen?” checks, because dict/set lookups are hash-table based and typically O(1) average-case, trading memory overhead for fast inserts and containment tests [Python Wiki: TimeComplexity; CPython setobject.c documentation references via Python docs]. This directly fits subscription ingestion, where repeated notifications, reconnect replays, or overlapping polling windows must be collapsed before downstream work.

3. Add a separate sorted/range index for replay and historical slicing.
Use a sorted structure for ordered scans across block numbers or event positions rather than forcing order into the hash index. SortedContainers is specifically optimized around a two-level list-of-sublists design with a max-index, using CPython list locality and lower pointer overhead than pointer-heavy tree nodes; it is designed for sorted iteration, bisect-style lookups, and range scans [Sorted Containers 2.4.0 documentation, Implementation Details; JOSS paper 'Python Sorted Containers']. That makes it the right secondary index for “which items fell in block range X to Y?” while the dict/set remains the fast dedupe front door.

4. Shape RPC requests around provider constraints, not around idealized client logic.
For read workflows, use web3.py batching because batching is allowed for reading transaction data, but do not use batch requests for sending new transactions [web3.py transactions documentation v7.1.0]. Cap any Alchemy batch at 1,000 requests because its docs state larger batches are likely to be less reliable [Alchemy Docs: Batch Requests]. Also do not assume batching is portable: QuickNode says batch requests are not recommended on Ethereum endpoints because they can slow dApp performance, and its WebSocket EVM endpoints do not follow the Ethereum JSON-RPC batch standard [QuickNode guide: How To Make Batch Requests on Ethereum; QuickNode Ethereum docs]. Therefore the system must make batching a provider-aware strategy toggle inside the web3.py access layer, not a universal default.

5. Convert throughput limits into explicit scheduling windows.
Alchemy measures throughput in CU/s at the account level with a 10-second rolling token-bucket window; a documented Free tier example is 500 CU/s, allowing up to 5,000 CUs across any 10-second window before 429s [Alchemy Docs: Throughput]. That means the hash index is not just for correctness; it is a throughput governor. Every duplicate request prevented by O(1)-average membership checks preserves CU budget in the rolling window [Python Wiki: TimeComplexity; CPython setobject.c documentation references via Python docs][Alchemy Docs: Throughput].

6. Shard historical work by block range because node products enforce range ceilings.
Chainstack documents that eth_getLogs range can be limited to 100 blocks on a Developer plan and 10,000 blocks on higher tiers [Chainstack Throughput Guidelines]. For deep historical state, archive access may be required: QuickNode documents archive-specific Erigon methods, and Infura positions archive access for historical Ethereum data older than the recent state window [QuickNode Ethereum API Overview; Infura Archive Access / Ethereum product pages]. Therefore the sorted/range index is operationally necessary: it lets the system split scans into plan-safe block windows, track completed ranges, and replay only missing intervals instead of re-querying entire histories.

7. Keep mutable in-memory indexes small; push cold numeric history to file-backed arrays only when access patterns are sequential.
NumPy memmap is suitable when datasets exceed RAM because it maps array-like numeric data onto disk, optimizing memory footprint and sequential/range access, but it is not a deduplicating or append-friendly hash structure [NumPy documentation: numpy.memmap]. So memmap can store cold block-number-aligned numeric artifacts, but it cannot replace the live dict/set dedupe layer. Likewise, pandas RangeIndex and IntervalIndex are useful indexing abstractions for compact regular ranges and interval membership/range selection, but they are not cheap append-heavy mutable stores [pandas 3.0.x documentation: Index objects; Advanced indexing]. They fit analysis outputs, not the hot ingestion path.

8. Use middleware only where transport-independent logic belongs.
web3.py middleware remains central for sanity checks, type conversion, ENS support, and transaction preparation; GasPriceStrategyMiddleware can inject gas price logic and estimate gas when omitted [web3.py 7.16.0 middleware docs]. In this system, middleware belongs at the request-shaping boundary, while dedupe, range partitioning, and provider-specific batching policy belong in the indexing and scheduling layers.

Interdependence proof:
- Remove data structure: The system loses O(1)-average deduplication and membership checks from dict/set, so repeated subscription events, overlapping block scans, and retry storms consume unnecessary RPC budget inside provider throughput windows [Python Wiki: TimeComplexity; CPython setobject.c documentation references via Python docs][Alchemy Docs: Throughput]. It also loses an efficient ordered secondary index for bisect/range scans, making plan-capped historical sharding much harder to track [Sorted Containers 2.4.0 documentation, Implementation Details; JOSS paper 'Python Sorted Containers'][Chainstack Throughput Guidelines].
- Remove web3.py: The system loses AsyncWeb3 async providers, IPC optimization, persistent-provider subscription support, middleware integration, and the documented read-batch interface [web3.py providers documentation v7.15.0/v7.0.0][web3.py Event Subscriptions documentation v7.15.0][web3.py 7.16.0 middleware docs][web3.py transactions documentation v7.1.0]. Then the data-structure design has no concrete Python transport stack in which to enforce provider-aware batching, subscriptions, or IPC-vs-HTTP execution choices.
- Remove Ethereum RPC performance: The system no longer has the governing reason to split hot hash indexes from ordered replay indexes. Without CU/s token-bucket limits, batch ceilings, provider-specific anti-batch guidance, archive-access boundaries, and method/range caps, the architecture collapses into a generic Python indexer instead of a performance-shaped RPC system [Alchemy Docs: Throughput][Alchemy Docs: Batch Requests][QuickNode guide: How To Make Batch Requests on Ethereum; QuickNode Ethereum docs][QuickNode Ethereum API Overview; Infura Archive Access / Ethereum product pages][Chainstack Throughput Guidelines].

Failure mode:
A client that treats all providers as batch-friendly will over-batch on Alchemy past 1,000 requests or use batching on QuickNode where it is not recommended, producing unreliable or slower behavior [Alchemy Docs: Batch Requests][QuickNode guide: How To Make Batch Requests on Ethereum; QuickNode Ethereum docs].
A client that replaces the hash dedupe layer with only range-oriented indexes will pay unnecessary RPC cost under rolling throughput limits because range indexes do not provide the same O(1)-average containment behavior as dict/set [Python Wiki: TimeComplexity; CPython setobject.c documentation references via Python docs][Alchemy Docs: Throughput].
