Title: Constraint-Native Contract Memory: a Solidity data-structure layer that forces LLM agents to operate only through bounded, benchmarkable state transitions

Thesis:
The right way to make LLM agents useful in Solidity is not to let them reason over arbitrary contract state, but to force them through a contract data-structure layer built from bounded append-only history, O(1) membership primitives, and transaction-scoped transient state, because Solidity’s gas, storage-layout, and compiler-bug constraints make unrestricted state access both expensive and unsafe [OpenZeppelin Contracts `Checkpoints.sol` (GitHub); OpenZeppelin community-contracts utils docs; Solidity security alert: `Transient Storage Clearing Helper Collision Bug`; Solidity security alert: `Inheritance Order Reversal On Storage End Warning Bug`; Ethereum.org `Smart contract security guidelines`; OpenAI: `Introducing EVMbench`].

The System:
Build one contract interaction system with three mandatory layers.

1. The persistent state layer is a constrained data-structure layer.
   - Historical facts are stored as append-only checkpoints, using the production pattern exemplified by OpenZeppelin’s `Checkpoints` library, which stores checkpoints keyed by block number and provides `upperLookupRecent` optimized toward recent history by starting near `len - sqrt(len)` instead of only searching from the start [OpenZeppelin Contracts `Checkpoints.sol` (GitHub)].
   - Dynamic membership uses enumerable sets or maps only for O(1) membership and length queries, because OpenZeppelin documents `contains` and length as efficient while warning that full ordered retrieval copies entire storage to memory, is “quite expensive,” and can make state-changing functions uncallable as the structure grows [OpenZeppelin community-contracts utils docs].
   - Destructive resets are forbidden as a normal operation, because `clear` is O(n), may exceed block gas limits, and deleting an `EnumerableSet` from storage may corrupt data; the documented safe pattern is element-by-element removal or fresh-instance replacement [OpenZeppelin community-contracts utils docs].

2. The Solidity implementation layer turns those data-structure constraints into hard interface rules.
   - The contract exposes no unbounded enumeration path to the agent. This is necessary because on-chain recomputation of ordered ownership-scale data becomes prohibitive at NFT-like scale: OpenZeppelin maintainers quantify 10,000 storage reads at about 2,100 gas each, or about 21 million gas before loop overhead, just to read owners for a 10k collection [OpenZeppelin Forum: `ERC721Enumerable gas profiling and optimizations`].
   - Transaction-scoped planning state is placed in transient storage via `TSTORE`/`TLOAD`, where both operations cost 100 gas, making it suitable for cheap per-transaction coordination such as locks or temporary key-value state [Solidity blog: `Transient Storage Opcodes in Solidity 0.8.24`; EIP-1153].
   - That transient layer is treated as a narrow scratchpad, not as durable logic, because Solidity’s own guidance warns advanced transient-storage use can break composability, and compiler warnings are currently triggered when `tstore` is used in assembly [Solidity blog: `Transient Storage Opcodes in Solidity 0.8.24`; EIP-1153].
   - Storage layout is explicitly pinned and audited because Solidity states storage layout is part of the language’s external interface, and recent releases added custom storage layouts in 0.8.29 and extended them in 0.8.31 [Solidity 0.8.29 release announcement; Solidity 0.8.31 release announcement; Solidity docs `Layout of State Variables in Storage and Transient Storage`].
   - Compiler version selection is a first-class design constraint, not a deployment afterthought, because there is a high-severity transient-storage clearing bug affecting 0.8.28–0.8.33 with `--via-ir`, fixed in 0.8.34, and a custom-storage-layout warning bug introduced in 0.8.29 and fixed in 0.8.36 [Solidity security alert: `Transient Storage Clearing Helper Collision Bug`; Solidity security alert: `Inheritance Order Reversal On Storage End Warning Bug`].
   - The agent-facing ABI also avoids assumptions based on storage packing from memory behavior, because Solidity docs emphasize memory layout differs from storage layout and memory arrays occupy multiples of 32 bytes except packed `bytes` and `string` [Solidity docs `Layout in Memory`].

3. The LLM-Agent layer is redesigned as a bounded-state operator rather than a free-form coder.
   - The agent receives a structured state schema matching the contract’s bounded primitives: recent checkpoint queries, O(1) membership tests, and ephemeral transient scratch slots.
   - Its job is to produce actions only within those constraints, then validate them through deterministic replay environments like EVMbench, whose harness deploys contracts, replays transactions deterministically, restricts unsafe RPC methods, and isolates exploit tasks in local Anvil environments [OpenAI: `Introducing EVMbench`].
   - This agent must be evaluated for patching and auditing, not just generation, because EVMbench includes Detect, Patch, and Exploit modes and uses custom graders plus red-teamed anti-cheating environments [OpenAI: `Introducing EVMbench`].
   - This redesign is necessary because recent benchmark evidence shows raw Solidity generation is structurally unreliable: SmartEval reports logic omissions at 35.3%, state-transition errors at 23.4%, and degradation as specification complexity rises across 9,000 generated contracts [arXiv: `SmartEval: A Benchmark for Evaluating LLM-Generated Smart Contracts from Natural Language Specifications`].
   - The implementation flow should therefore be multi-agent and provenance-carrying, using a pipeline style like the 2026 agentic workflow that parses text into structured schemas, generates Solidity, runs compilation and security checks, and iteratively refines outputs with provenance metadata [arXiv: `An end-to-end agentic pipeline for smart contract translation and quality evaluation`].
   - Final safety checking should not trust the model’s judgment alone; it should require typed specifications, type checking, and concrete execution to produce validated counterexamples in the style of Neuroforger’s certified violation witnesses [arXiv: `Neuroforger: certified violation witnesses for smart contracts verification via LLMs`].

This creates one novel system: a Constraint-Native Contract Memory layer where data-structure choices define the only legal state access patterns, Solidity compiler and storage rules harden those patterns into safe interfaces, and the LLM agent is reduced from “author of arbitrary contract behavior” to “planner over bounded state transitions.” Every keyword is structural: the data structure defines the reachable state, Solidity defines the safe realization, and the LLM agent defines the operational loop that exploits those constraints instead of fighting them.

Interdependence proof:
- Remove data structure: The system collapses into unconstrained state access. The agent would need full retrieval or recomputation paths that OpenZeppelin documents as expensive and potentially uncallable at scale, and historical access would lose the append-only checkpoint optimization that makes recent-state lookup practical [OpenZeppelin community-contracts utils docs; OpenZeppelin Contracts `Checkpoints.sol` (GitHub); OpenZeppelin Forum: `ERC721Enumerable gas profiling and optimizations`].
- Remove solidity: The system loses its real constraints and becomes an abstract planner with no guarantee that storage layout, transient storage, memory behavior, or compiler bugs are handled correctly. That breaks the central safety claim because storage layout is part of Solidity’s external interface, transient-storage use has known composability caveats, and current compiler advisories directly affect storage and transient storage correctness [Solidity docs `Layout of State Variables in Storage and Transient Storage`; Solidity blog: `Transient Storage Opcodes in Solidity 0.8.24`; Solidity security alert: `Transient Storage Clearing Helper Collision Bug`; Solidity security alert: `Inheritance Order Reversal On Storage End Warning Bug`].
- Remove LLM-Agent: The system becomes only a manual contract design pattern. It loses the benchmarked detect/patch/exploit loop, deterministic replay, anti-cheating evaluation, iterative generation-and-check workflow, and witness-based verification that are needed precisely because unconstrained Solidity generation has high rates of logic omissions and state-transition errors [OpenAI: `Introducing EVMbench`; arXiv: `SmartEval: A Benchmark for Evaluating LLM-Generated Smart Contracts from Natural Language Specifications`; arXiv: `An end-to-end agentic pipeline for smart contract translation and quality evaluation`; arXiv: `Neuroforger: certified violation witnesses for smart contracts verification via LLMs`].

Failure mode:
If the agent is allowed one unbounded enumeration endpoint, it will route around the bounded-state design and reintroduce gas-scaled failure modes that OpenZeppelin already documents as expensive or uncallable [OpenZeppelin community-contracts utils docs].
If the contract uses transient storage on affected compiler versions or careless custom layouts, the “safe scratchpad” premise fails at compilation level before agent logic is even evaluated [Solidity security alert: `Transient Storage Clearing Helper Collision Bug`; Solidity security alert: `Inheritance Order Reversal On Storage End Warning Bug`].

Open question:
Can an EVMbench-style harness be extended to score whether an LLM agent stays entirely within a formally declared bounded data-structure interface—checkpoint lookups, O(1) membership queries, and transient scratch state—while Neuroforger-style witnesses prove every rejected action would have violated those interface constraints [OpenAI: `Introducing EVMbench`; arXiv: `Neuroforger: certified violation witnesses for smart contracts verification via LLMs`]?
