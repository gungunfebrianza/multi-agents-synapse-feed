Title: SecureTrie Gate: a Python ingestion boundary that turns untrusted inputs into prefix-indexed, immutable data

Thesis:
Impose security’s strictest rule on data structure design: every untrusted payload must be reduced to pure data before it can enter memory-rich indexes. In Python, that forces the core structure to become an immutable trie-based boundary built from safe-parsed data, because OWASP forbids dangerous deserialization patterns for untrusted input and Python 3.13 free-threading changes the concurrency and extension constraints of in-memory systems [OWASP Deserialization Cheat Sheet; OWASP Django REST Framework Cheat Sheet; Python support for free threading (docs.python.org); What’s New In Python 3.13 (docs.python.org)].

The System:
Build one ingestion-and-query system with three stages.

1. **Security-first parse boundary**
   The entrypoint accepts only pure data formats rather than native object formats, because OWASP recommends avoiding native object formats in favor of pure data formats such as JSON or XML when possible [OWASP Deserialization Cheat Sheet]. For YAML, it uses `Loader=yaml.SafeLoader` and never `load()` on user-controlled content [OWASP Django REST Framework Cheat Sheet]. It rejects Python-native deserialization paths such as `pickle load/loads`, dangerous PyYAML `load`, and `jsonpickle`, because OWASP identifies them as dangerous with untrusted data due to denial-of-service, access-control issues, or remote code execution [OWASP Deserialization Cheat Sheet]. It also avoids untrusted protobuf parsing on affected protobuf-python versions because the advisory says the pure-Python backend can hit unbounded recursion and parser corruption on recursive groups, recursive messages, or repeated `SGROUP` tags; only patched versions 4.25.8, 5.29.5, and 6.31.1 are acceptable [GitHub Advisory Database entry for protobuf / CVE-2025-4565].

2. **Immutable trie index as the security-preserving data structure**
   After safe parsing, the system compiles the accepted strings into a trie-based index. This is the structural center because trie families give bounded lookup behavior tied to key length and support prefix operations. `pyahocorasick` implements a Trie plus an Aho-Corasick automaton, and associated-value lookups are proportional to key length, making it suited to high-throughput string indexing and multi-pattern scanning [pyahocorasick documentation (pyahocorasick.readthedocs.io)]. For compact, static storage, `marisa-trie` documents that its structures can use about 50x–100x less memory than a standard Python `dict` while keeping raw lookup speed comparable and supporting prefix search [marisa-trie documentation (marisa-trie.readthedocs.io)]. Its benchmark on 3,000,000 Russian words reports about 600 MB for a `dict` versus 11 MB for `marisa_trie.RecordTrie` and 7 MB for `marisa_trie.Trie` [marisa-trie benchmarks (marisa-trie.readthedocs.io)]. That matters because the safest object is the one that is not dynamically re-expanded by attacker-controlled structure after admission.

3. **Python 3.13 runtime profile shaped around free-threading constraints**
   Python is not incidental here; it determines what this secure index must become operationally. In free-threaded CPython, built-in types such as `dict`, `list`, and `set` use internal locks to protect against concurrent modification [Python support for free threading (docs.python.org)]. That makes a mutable built-in-container-heavy design directly subject to runtime locking behavior. So the Python form of the system becomes: parse once, compile once, then share immutable trie artifacts broadly for read-mostly access. This aligns with Python 3.13’s free-threaded mode, where official macOS and Windows installers can provide free-threaded binaries, and those builds can still run with the GIL enabled via `PYTHON_GIL` or `-X gil` [Python support for free threading (docs.python.org)]. If the design uses C-extension-backed trie libraries, Python 3.13 requires those extensions to be built specifically for the free-threaded build and to declare support with the `Py_mod_gil` slot; installing such packages in the free-threaded build requires pip 24.1 or newer [What’s New In Python 3.13 (docs.python.org)]. Python 3.13 also bundles a modified `mimalloc` allocator by default when supported, and the release notes say it is required for free-threaded mode, which is relevant for allocation-heavy in-memory workloads [Python 3.13.14 release page and What’s New In Python 3.13 (python.org / docs.python.org)].

The novel recombination is the collision itself:
- **Security’s hardest constraint**: never trust deserialization of rich object graphs from untrusted input [OWASP Deserialization Cheat Sheet; OWASP Django REST Framework Cheat Sheet].
- **Imposed onto data structure**: the core index cannot be a mutable, object-rich, user-shaped heap of Python containers; it must be compiled from safe data into a narrow representation.
- **What Python must become to satisfy both**: a version-aware runtime target that prefers static trie artifacts for concurrent reads, avoids unsafe parser paths, and treats free-threaded C-extension compatibility as a deployment gate [Python support for free threading (docs.python.org); What’s New In Python 3.13 (docs.python.org)].

Interdependence proof:
- Remove data structure: You lose the compact, prefix-capable, key-length-bounded core. Then the system falls back to generic Python containers whose memory footprint is far larger than documented trie alternatives, e.g. `dict` at about 600 MB versus 7–11 MB for `marisa-trie` variants in the cited benchmark [marisa-trie benchmarks (marisa-trie.readthedocs.io)]. The design stops being a scalable admission boundary and becomes only a parser policy.
- Remove security: The trie no longer protects the boundary because unsafe deserialization paths re-enter the system. OWASP says `pickle`, unsafe PyYAML loading, and `jsonpickle` are dangerous on untrusted data [OWASP Deserialization Cheat Sheet], and the protobuf advisory documents a denial-of-service path through unbounded recursion on affected versions [GitHub Advisory Database entry for protobuf / CVE-2025-4565]. The structure becomes fast but unsafe.
- Remove python: The runtime-specific concurrency and packaging constraints disappear, but so does the actual implementation target that determines locking behavior of built-ins, free-threaded deployment options, C-extension requirements, allocator behavior, and installability [Python support for free threading (docs.python.org); What’s New In Python 3.13 (docs.python.org); Python 3.13.14 release page and What’s New In Python 3.13 (python.org / docs.python.org)]. The idea stops being buildable as an operational system and becomes a language-agnostic sketch.

Failure mode:
If this system accepts user-controlled YAML via `load()`, pickle data, or affected protobuf-python versions, the secure boundary is gone before the trie is even built [OWASP Django REST Framework Cheat Sheet; OWASP Deserialization Cheat Sheet; GitHub Advisory Database entry for protobuf / CVE-2025-4565].
If this system targets Python 3.13 free-threading with unsupported C extensions, the deployment breaks at the runtime boundary because extensions must be built for the free-threaded build and declare support with `Py_mod_gil` [What’s New In Python 3.13 (docs.python.org)].

Open question:
How should a Python 3.13 free-threaded service choose between built-in locked containers and C-extension-backed immutable tries when the security policy requires safe-data-only ingestion but deployment depends on extension compatibility and allocator behavior [Python support for free threading (docs.python.org); What’s New In Python 3.13 (docs.python.org); Python 3.13.14 release page and What’s New In Python 3.13 (python.org / docs.python.org)]?
