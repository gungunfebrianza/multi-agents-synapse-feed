Title: Browser-Native Transaction Firebreak: a security extension that classifies smart-contract approvals before signature

Thesis:
The strongest buildable defense against wallet drainers is a browser extension that performs pre-signature smart-contract risk classification locally, because the attack surface is the browser-mediated signing moment, the dominant theft pattern is approval and signing abuse rather than key theft, and modern extension rules plus review policies force the detector to be package-contained, reviewable, and narrowly single-purpose [Chrome for Developers: Manifest V3 (developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3); Chrome Web Store Program Policies: Additional Requirements for Manifest V3 (developer.chrome.com/docs/webstore/program-policies/mv3-requirements); Chainalysis 2024 Crypto Crime Report PDF (chainalysis.com/wp-content/uploads/2024/06/the-2024-crypto-crime-report-release.pdf); Chainalysis blog: Understanding Crypto Drainers (chainalysis.com/blog/crypto-drainers/); EIP-2612 (eips.ethereum.org/EIPS/eip-2612); EIP-712: Typed structured data hashing and signing (eips.ethereum.org/EIPS/eip-712)].

The System:
Build a browser extension whose only purpose is to intercept wallet-adjacent signing context and convert raw transaction or typed-data prompts into a hard risk verdict before the user signs.

The browser-extension layer is structurally necessary because wallet-drainer campaigns operate by pushing victims from phishing pages and compromised high-trust social accounts into browser wallet flows where they connect a wallet and sign a transaction [Google Cloud Blog / Mandiant: CLINKSINK drainer campaigns (cloud.google.com/blog/topics/threat-intelligence/solana-cryptocurrency-stolen-clinksink-drainer-campaigns/); Chainalysis 2025 Crypto Crime Report materials (chainalysis.com/wp-content/uploads/2025/03/the-2025-crypto-crime-report-release.pdf)]. The extension must be designed for Manifest V3, which replaces persistent background pages with service workers and forbids remotely hosted executable code; only JavaScript included in the extension package may execute, and Chrome Web Store policy requires submitted functionality to be reviewable from that code [Chrome for Developers: Manifest V3 (developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3); Chrome Web Store Program Policies: Additional Requirements for Manifest V3 (developer.chrome.com/docs/webstore/program-policies/mv3-requirements)]. That constraint dictates the architecture: packaged decoders, packaged heuristics, and no remote code path for transaction interpretation. The same review pressure is reinforced by Mozilla’s requirement for reviewable source code and bundled dependencies or build-time retrieval only through official package managers [Mozilla Add-ons Community Blog: Updated Add-on policies — simplified, clarified (blog.mozilla.org/addons/2025/06/23/updated-add-on-policies-simplified-clarified/)].

The security layer is the decision engine. Its threat model is approval phishing and drainer behavior. Chainalysis documented that victims are commonly tricked into granting spend approvals instead of surrendering keys, and reported one approval-phishing address likely stole $44.3 million from thousands of victims between May 2022 and November 2023 [Chainalysis 2024 Crypto Crime Report PDF (chainalysis.com/wp-content/uploads/2024/06/the-2024-crypto-crime-report-release.pdf)]. Mandiant documented a concrete drainer flow in which victims connected wallets and were prompted to sign a transaction that enabled theft [Google Cloud Blog / Mandiant: CLINKSINK drainer campaigns (cloud.google.com/blog/topics/threat-intelligence/solana-cryptocurrency-stolen-clinksink-drainer-campaigns/)]. Browser-native security matters because Chrome already deploys real-time Safe Browsing checks and Enhanced Protection that includes extra protection from malicious extensions, showing that browser-side controls are a live defensive layer at internet scale [Google Chrome Blog: real-time Safe Browsing protections (blog.google/products-and-platforms/products/chrome/google-chrome-safe-browsing-real-time/); Google Online Security Blog March 14 2024 (security.googleblog.com/2024/03/blog-post.html); Chrome Help: Safe Browsing protection level (support.google.com/chrome/answer/9890866)].

The smart-contract layer is what turns generic browser security into a useful verdict. The extension should classify three signing surfaces.

First, calldata transactions. The extension parses the first 4 bytes of calldata to identify candidate functions, using selector analysis and a signature database such as 4byte.directory for human-readable mapping [ethereum.org: Transactions (ethereum.org/developers/docs/transactions); 4byte Directory docs (4byte.directory/docs/)]. It then simulates the transaction before signature using modern simulation APIs that expose asset changes, internal calls, decoded traces, and logs [Alchemy Docs: Transaction Simulation (alchemy.com/docs/reference/simulation)]. This lets the extension show the user not “contract interaction” but “this transaction causes asset movement to X through Y internal call path.”

Second, typed-data signatures. The extension parses EIP-712 fields, especially chainId and verifyingContract, to identify the target contract and message type before signature [EIP-712: Typed structured data hashing and signing (eips.ethereum.org/EIPS/eip-712)]. This is essential because permit-style approvals are a major risk surface: ERC-2612 standardizes signed approvals via permit, and Permit2 extends signature-based transfers and time-bound allowances, allowing spending authorization without a traditional on-chain approve first [EIP-2612 (eips.ethereum.org/EIPS/eip-2612); Uniswap Developers: Permit2 Overview and Allowance Transfer (developers.uniswap.org/docs/protocols/permit2/overview, developers.uniswap.org/docs/protocols/permit2/concepts/allowance-transfer)]. A security tool that ignores signed-message approvals misses the exact mechanism attackers now exploit.

Third, upgradeable contract indirection. The extension checks ERC-1967 storage slots to determine implementation addresses for a large class of proxies, using the standard implementation slot and monitoring for Upgraded events as defined by the standard [EIP-1967: Proxy Storage Slots (eips.ethereum.org/EIPS/eip-1967)]. That matters because simulation and selector decoding against the proxy address alone can hide the real execution target; proxy resolution makes the risk label attach to the implementation that will actually run.

The system’s adversarial recombination is this: use browser-extension constraints to force a security engine that cannot rely on remote executable updates, then use smart-contract introspection to make that constrained engine precise enough to stop approval and drainer abuse. Browser extension rules shape deployment, security defines the objective, and smart-contract analysis supplies the only usable signal.

Interdependence proof:
- Remove browser extension: The system loses the signing-moment control point. Safe Browsing and store review are browser-side controls, and drainer campaigns succeed by moving users through phishing pages into wallet prompts in the browser [Google Chrome Blog: real-time Safe Browsing protections (blog.google/products-and-platforms/products/chrome/google-chrome-safe-browsing-real-time/); Google Cloud Blog / Mandiant: CLINKSINK drainer campaigns (cloud.google.com/blog/topics/threat-intelligence/solana-cryptocurrency-stolen-clinksink-drainer-campaigns/)]. Without the extension, there is no packaged, reviewable, client-side enforcement surface governed by MV3 rules [Chrome for Developers: Manifest V3 (developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3)].
- Remove security: The system becomes a decoder without purpose. Calldata parsing, EIP-712 parsing, simulation, and proxy detection would produce descriptions, but not a defense against approval phishing and drainer workflows that have already caused large-scale theft [Chainalysis 2024 Crypto Crime Report PDF (chainalysis.com/wp-content/uploads/2024/06/the-2024-crypto-crime-report-release.pdf); Google Cloud Blog / Mandiant: CLINKSINK drainer campaigns (cloud.google.com/blog/topics/threat-intelligence/solana-cryptocurrency-stolen-clinksink-drainer-campaigns/)].
- Remove smart contract: The system cannot distinguish benign dapp interaction from permit-style or approval-style theft. Attackers exploit transaction signing and signature-based approvals; those require calldata selector analysis, EIP-712 typed-data parsing, simulation of asset changes, and proxy resolution to classify correctly [ethereum.org: Transactions (ethereum.org/developers/docs/transactions); EIP-712: Typed structured data hashing and signing (eips.ethereum.org/EIPS/eip-712); EIP-2612 (eips.ethereum.org/EIPS/eip-2612); Uniswap Developers: Permit2 Overview and Allowance Transfer (developers.uniswap.org/docs/protocols/permit2/overview, developers.uniswap.org/docs/protocols/permit2/concepts/allowance-transfer); Alchemy Docs: Transaction Simulation (alchemy.com/docs/reference/simulation); EIP-1967: Proxy Storage Slots (eips.ethereum.org/EIPS/eip-1967)].

Failure mode:
An attacker can still present a transaction whose malicious effect is socially disguised even after correct decoding, because the final action is a user signature on a browser wallet prompt [Google Cloud Blog / Mandiant: CLINKSINK drainer campaigns (cloud.google.com/blog/topics/threat-intelligence/solana-cryptocurrency-stolen-clinksink-drainer-campaigns/)].
An extension that depends on broad interception of network flows will also fail under MV3 design pressure, because Chrome moved common interception patterns away from blocking webRequest listeners toward declarativeNetRequest rules [Chrome for Developers: Replace blocking web request listeners (developer.chrome.com/docs/extensions/develop/migrate/blocking-web-requests)].

Open question:
How far can a fully packaged MV3 extension push local smart-contract risk classification for EIP-712, Permit2, proxy contracts, and simulated asset changes before store-review constraints and no-remote-code rules force accuracy tradeoffs that attackers can systematically exploit [Chrome for Developers: Manifest V3 (developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3); Chrome Web Store Program Policies: Additional Requirements for Manifest V3 (developer.chrome.com/docs/webstore/program-policies/mv3-requirements); EIP-712: Typed structured data hashing and signing (eips.ethereum.org/EIPS/eip-712); Uniswap Developers: Permit2 Overview and Allowance Transfer (developers.uniswap.org/docs/protocols/permit2/overview, developers.uniswap.org/docs/protocols/permit2/concepts/allowance-transfer); Alchemy Docs: Transaction Simulation (alchemy.com/docs/reference/simulation)]
