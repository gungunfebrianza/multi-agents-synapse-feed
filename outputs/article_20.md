Title: A Browser Extension Should Be a Smart-Contract Firewall, Not a Wallet Add-On

Thesis:
The right buildable defense against Web3 draining attacks is a browser extension that acts as a pre-sign smart-contract firewall: it classifies approvals, signatures, proxy upgrades, and simulated transaction effects before wallet confirmation, because browser extensions are the only user-local layer positioned between dapp pages and irreversible smart-contract actions, and security only works here if it is contract-aware and implementable within MV3 constraints [Chrome for Developers: Manifest V3 (developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3); Chrome for Developers: Deal with remote hosted code violations; EIP-2612; EIP-1271; OpenZeppelin Docs: Proxy API; EIP-1967; Chainalysis 2024 Crypto Crime Report PDF].

The System:
Build a browser extension that treats every high-risk Web3 interaction as a smart-contract security event, then enforces a local verdict pipeline before the wallet confirmation step.

The browser-extension layer is structurally necessary because implementable defenses already exist there. Wallet Guard states that it injects the Ethereum web3 API into webpages to securely read and verify transaction requests from dapps, and combines transaction simulation with proactive phishing detection [GitHub: wallet-guard/wallet-guard-extension]. That model fits the extension form factor: it can observe dapp context in the browser, inspect requests before signing, and present warnings in the same user session.

The security core should prioritize approval phishing first. Chainalysis reports that the most successful approval-phishing address stole $44.3 million from thousands of victims between May 2022 and November 2023 [Chainalysis 2024 Crypto Crime Report PDF]. That makes spender approvals and approval-like signatures the primary detection target, not a secondary feature.

The smart-contract layer must classify more than raw transactions:
- ERC-2612 permit requests are approval-risk events because permit(owner, spender, value, deadline, v, r, s) sets allowance by signature rather than by an on-chain approve from msg.sender [EIP-2612].
- ERC-1271 must be supported because contract wallets validate signatures through isValidSignature(hash, signature), returning magic value 0x1626ba7e on success [EIP-1271].
- Proxy upgrade paths must be flagged as high risk because OpenZeppelin documents that ERC1967 proxies store implementation in the EIP-1967 slot, UUPSUpgradeable exposes upgradeToAndCall, and TransparentUpgradeableProxy has built-in admin and upgrade interfaces [OpenZeppelin Docs: Proxy API; EIP-1967].
- Selector coverage must be current because OpenZeppelin’s v5.1.0 changelog states that UUPSUpgradeable, TransparentUpgradeableProxy, and ProxyAdmin removed upgradeTo/upgrade in favor of upgradeToAndCall/upgradeAndCall behavior changes [OpenZeppelin Contracts 5.x changelog].

A buildable detection pipeline already exists from available APIs. GoPlus provides Phishing Site Detection API, Signature Data Decode API, dApp Security Info API, Approval Security API, and EVM Transaction Simulation API [GoPlus Docs: Introduction; API Overview]. Its approval-security responses expose whether the approved contract is open source, whether it is flagged as malicious, token/NFT approval details, and for approval-management endpoints the approval date, allowance amount, and transaction ID [GoPlus Docs: Token Approval Security response details; ERC-20/ERC-721/ERC-1155 approval security endpoints]. That enables one unified verdict engine:
1. Check the page/domain with phishing-site and dapp-security signals [GoPlus Docs: Introduction; API Overview].
2. Decode the signature or calldata [GoPlus Docs: Introduction; API Overview].
3. Reclassify permit as approval risk, not harmless signing [EIP-2612].
4. Support ERC-1271 account flows instead of assuming all signatures are from EOAs [EIP-1271].
5. Flag proxy admin changes and upgrade entrypoints as critical-risk actions [OpenZeppelin Docs: Proxy API; EIP-1967; OpenZeppelin Contracts 5.x changelog].
6. Simulate the transaction before signing [GoPlus Docs: API Overview; Introduction].
7. Surface stale unlimited approvals or suspicious spenders using allowance amount, approval date, and spender risk metadata [GoPlus Docs: Token Approval Security response details; ERC-20/ERC-721/ERC-1155 approval security endpoints].

The architecture must be designed around extension-platform constraints, not against them. Chrome MV3 removes remotely hosted code, allowing execution only of JavaScript bundled inside the extension package [Chrome for Developers: Manifest V3 (developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3); Chrome for Developers: Deal with remote hosted code violations]. That means the extension cannot hot-load remote detection logic into its runtime. Chrome also limits blocking webRequest in MV3; for consumer extensions, request blocking must usually use declarativeNetRequest instead of JavaScript interception [Chrome for Developers: chrome.webRequest API (developer.chrome.com/docs/extensions/reference/api/webRequest); Chrome for Developers: chrome.declarativeNetRequest API]. So the firewall should not depend on arbitrary request interception as its main enforcement primitive. It should depend on packaged classification logic plus API-backed risk lookups and simulation.

Distribution and permission scope are also part of the design. Broad host permissions like <all_urls> and sensitive permissions such as cookies or webRequest trigger longer/manual Chrome Web Store review [Chrome for Developers: Chrome Web Store review process]. Cryptocurrency is also listed among categories not currently featured in the store [Chrome for Developers: Chrome Web Store Program Policies]. So the product must assume review friction and weak store discoverability. On Firefox, host_permissions grant access for cookies, webRequest, tabs metadata, and script injection on matching hosts, users can grant or revoke those permissions ad hoc, and declarativeNetRequest support remains more limited and conditional than Chrome’s [MDN: host_permissions; MDN: permissions; MDN: declarativeNetRequest]. MDN also states that host permissions do not let MV3 content scripts perform cross-origin fetches without server CORS opt-in, pushing security logic toward extension pages or background contexts rather than page scripts alone [MDN: Content scripts]. That directly shapes implementation: the page hook reads requests, but core analysis belongs in extension-controlled contexts.

Interdependence proof:
- Remove browser extension: The system loses the user-local inspection point that can read dapp-originated requests in-session and combine phishing detection with transaction verification, which Wallet Guard demonstrates at the extension layer [GitHub: wallet-guard/wallet-guard-extension]. The result is no practical place to intercept signing intent before wallet confirmation.
- Remove security: The system becomes a passive smart-contract decoder with no adversarial purpose. That fails the actual threat model, where approval phishing alone has already produced $44.3 million in theft from one address cluster [Chainalysis 2024 Crypto Crime Report PDF]. Without phishing detection, approval-risk scoring, malicious-contract checks, and simulation, the extension is just a UI shell.
- Remove smart contract: The system cannot distinguish harmless browsing from irreversible approval and upgrade risk. It misses that ERC-2612 permit is an approval event [EIP-2612], misses ERC-1271 contract-wallet flows [EIP-1271], and misses proxy upgrade/admin danger paths including current upgradeToAndCall and upgradeAndCall entrypoints [OpenZeppelin Docs: Proxy API; EIP-1967; OpenZeppelin Contracts 5.x changelog]. Without contract semantics, “security” alerts collapse into generic phishing warnings.

Failure mode:
A malicious flow can still degrade if the extension over-relies on network interception, because MV3 restricts blocking webRequest for consumer extensions and remote code loading is disallowed [Chrome for Developers: chrome.webRequest API (developer.chrome.com/docs/extensions/reference/api/webRequest); Chrome for Developers: Manifest V3 (developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3)].
A malicious flow can also evade weak classifiers if the extension treats permit signatures, ERC-1271 validations, or current proxy upgrade entrypoints as ordinary interactions instead of high-risk smart-contract actions [EIP-2612; EIP-1271; OpenZeppelin Contracts 5.x changelog].

Open question:
How should a browser-extension smart-contract firewall represent one unified risk model that correctly scores EOAs, ERC-1271 smart accounts, ERC-2612 permits, and evolving proxy upgrade entrypoints while staying fully compatible with MV3’s no-remote-code and limited interception rules [EIP-1271; EIP-2612; OpenZeppelin Contracts 5.x changelog; Chrome for Developers: Manifest V3 (developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3)]?
