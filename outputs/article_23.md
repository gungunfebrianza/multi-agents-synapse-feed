Title: MV3 Smart-Contract Firewall: a browser extension that treats every wallet prompt as a security-critical protocol object

Thesis:
The buildable opportunity is a browser extension that does not act like a wallet and does not try to intercept everything, but instead functions as an MV3-compatible smart-contract security firewall: it observes browser wallet RPC and signing flows, decodes modern Ethereum transaction and message formats, and issues risk verdicts at the exact point where users approve contract actions. This is the correct architecture because browser compromise and extension abuse are active attack patterns, modern Ethereum signing now spans typed transactions, EIP-712, and ERC-4337 flows, and Chrome/Firefox extension rules sharply constrain how inspection logic can run and ship [Chrome for Developers: Extensions / Manifest V3; Chrome for Developers: Deal with remote hosted code violations; Chrome for Developers: chrome.webRequest API; Chrome for Developers: chrome.declarativeNetRequest API; Microsoft Security Blog: Infostealers without borders: macOS, Python stealers, and platform abuse; Microsoft Security Blog: Dissecting Sapphire Sleet’s macOS intrusion from lure to compromise; Microsoft Security Intelligence: Behavior:MacOS/XCSSET.BA threat description; EIP-2718; EIP-2930; EIP-4844; EIP-1559; EIP-712; ERC-4337; ERC-7769; ERC-4337 Documentation].

The System:
The system is a packaged browser extension built for Manifest V3 constraints. On Chromium, that means its inspection engine runs from a service-worker-based architecture rather than a persistent background page, and its logic must be bundled in the extension package because remotely hosted code is prohibited and reviewed by the Chrome Web Store [Chrome for Developers: Extensions / Manifest V3; Chrome for Developers: Deal with remote hosted code violations]. That packaging requirement is a security feature: the signing-risk logic is inspectable at review time instead of being changeable from a remote server [Chrome for Developers: Deal with remote hosted code violations].

Its collection layer watches the surfaces where dapps and wallet bridges actually request signing. Those include methods such as eth_sendTransaction, personal_sign, and eth_signTypedData variants, which WalletConnect explicitly documents for EVM wallet session_request flows and which are also common in MetaMask-style browser signing [WalletConnect Docs: Web usage; WalletConnect Docs: EVM chain support; MetaMask docs search results]. On Chrome, the extension can observe and analyze requests with chrome.webRequest when it has the webRequest permission and matching host permissions, but arbitrary blocking via webRequestBlocking is not available to most consumer extensions, so network-side intervention must generally be expressed through declarativeNetRequest where applicable [Chrome for Developers: chrome.webRequest API; Chrome for Developers: chrome.declarativeNetRequest API]. On Firefox, host_permissions govern access to webRequest, tabs, cookies, and script injection, and declarativeNetRequest works within the host access the user grants [MDN: host_permissions; MDN: permissions; MDN: declarativeNetRequest].

Its parser is smart-contract-specific. It must decode EIP-2718 typed envelopes because later Ethereum transaction formats are wrapped as typed transactions, including EIP-2930 access-list transactions and EIP-4844 blob transactions [EIP-2718; EIP-2930; EIP-4844]. It must interpret EIP-1559 fee fields such as max priority fee and max fee because modern approval risk depends on fee ceilings and replacement behavior, not only legacy gasPrice semantics [EIP-1559]. It must parse EIP-712 typed structured data because many dangerous approvals and permissions are presented as structured sign requests defined by types, domain separator parameters, and a message object [EIP-712]. It must also understand ERC-4337 UserOperation flows and the ERC-7769 JSON-RPC surface because account-abstraction activity no longer appears only as legacy Ethereum transactions [ERC-4337; ERC-7769; ERC-4337 Documentation].

Its security model is adversarial by design. Microsoft documented malware campaigns that steal browser passwords, cryptocurrency wallet information, cloud credentials, and developer secrets, including lures aimed at browser-based crypto users [Microsoft Security Blog: Infostealers without borders: macOS, Python stealers, and platform abuse]. Microsoft also documented a macOS intrusion that targeted nine crypto wallet extensions and exfiltrated browser sign-in data, cookies, and keychain databases, with mitigations that included enforcing hardware wallet policies and rotating browser-stored credentials [Microsoft Security Blog: Dissecting Sapphire Sleet’s macOS intrusion from lure to compromise]. Microsoft’s XCSSET description adds clipboard manipulation, unexpected browser redirects, login-page appearance changes, and unusual wallet-extension behavior as live symptoms in crypto workflows [Microsoft Security Intelligence: Behavior:MacOS/XCSSET.BA threat description]. Google’s fraud advisory adds “invisible pages” used to host malicious instructions and phishing pages that bypass standard web filters [Google Blog: June 2026 frauds and scams advisory]. Together these facts define the extension’s role: not merely decoding contract data, but correlating sign requests with hostile browser context.

That leads to the core product behavior: before a user confirms a transaction or typed-data signature, the extension generates a local verdict from three inputs:
1. the signing primitive requested by the page or wallet bridge, such as eth_sendTransaction, personal_sign, or eth_signTypedData [WalletConnect Docs: Web usage; WalletConnect Docs: EVM chain support; MetaMask docs search results];
2. the exact smart-contract object being approved, decoded across EIP-2718, EIP-1559, EIP-712, and ERC-4337/7769 forms [EIP-2718; EIP-2930; EIP-4844; EIP-1559; EIP-712; ERC-4337; ERC-7769; ERC-4337 Documentation];
3. browser-side danger signals consistent with documented attacks, including suspicious redirects, page-level manipulation symptoms, extension trust concerns, and wallet-phishing patterns [Microsoft Security Intelligence: Behavior:MacOS/XCSSET.BA threat description; Chrome Web Store Help: Install and manage extensions; Google Chrome Community: Fake Chrome extensions; Google Blog: June 2026 frauds and scams advisory].

Because Chromium store policy is tightening around wallet-like extensions, including compliance reviews, bans on alternative remote logic execution, and Edge requirements that crypto-wallet extensions be published only by verified company accounts with only essential permissions, the product should ship as a companion analysis extension rather than a replacement wallet [Chrome Web Store Program Policies; Chrome Web Store review process; Microsoft Learn: Developer policies for the Microsoft Edge Add-ons store]. That positioning is structurally important: the extension can stay within essential permissions, keep all logic packaged, and avoid becoming another high-value wallet target while still analyzing the exact approval surface where smart-contract risk materializes [Chrome Web Store Program Policies; Microsoft Learn: Developer policies for the Microsoft Edge Add-ons store].

Interdependence proof:
- Remove browser extension: the system loses the only place where it can directly observe browser wallet RPC requests, host-page context, extension trust signals, redirects, and DOM-adjacent phishing symptoms at approval time. The result is no real-time access to eth_sendTransaction, personal_sign, eth_signTypedData flows inside the browsing session, and no MV3-packaged local enforcement model [WalletConnect Docs: Web usage; WalletConnect Docs: EVM chain support; MetaMask docs search results; Chrome for Developers: Extensions / Manifest V3; MDN: host_permissions].
- Remove security: the system collapses into a decoder. It can still parse EIP-2718, EIP-1559, EIP-712, and ERC-4337 objects, but it no longer addresses the documented threats that actually compromise users: infostealers, wallet-extension targeting, clipboard hijacking, fake wallet extensions, redirects, and phishing infrastructure [Microsoft Security Blog: Infostealers without borders: macOS, Python stealers, and platform abuse; Microsoft Security Blog: Dissecting Sapphire Sleet’s macOS intrusion from lure to compromise; Microsoft Security Intelligence: Behavior:MacOS/XCSSET.BA threat description; Chrome Web Store Help: Install and manage extensions; Google Chrome Community: Fake Chrome extensions; Google Blog: June 2026 frauds and scams advisory].
- Remove smart contract: the system loses the ability to distinguish harmless browser activity from irreversible blockchain approvals. Without EIP-2718 transaction parsing, EIP-1559 fee interpretation, EIP-712 structured-sign decoding, and ERC-4337 UserOperation handling, it cannot explain or score what the user is actually authorizing [EIP-2718; EIP-2930; EIP-4844; EIP-1559; EIP-712; ERC-4337; ERC-7769; ERC-4337 Documentation].

Failure mode:
An attacker who compromises the host before approval can still present a convincing dapp flow while steering the user into a legitimate-looking sign request; the extension only works if its local packaged logic and granted permissions remain intact [Microsoft Security Blog: Dissecting Sapphire Sleet’s macOS intrusion from lure to compromise; Chrome for Developers: Deal with remote hosted code violations].  
A fake or untrusted extension that asks for a seed phrase bypasses contract analysis entirely, which is why the system must never request recovery phrases and must rely on browser trust protections and verified distribution practices [Chrome Web Store Help: Install and manage extensions; Google Chrome Community: Fake Chrome extensions].

Open question:
How should an MV3-packaged browser extension present one unified risk model across EIP-712 signatures, EIP-2718/EIP-1559 transactions, and ERC-4337 UserOperations when Chrome limits arbitrary in-flight blocking for consumer extensions but users still need a single approval-time verdict [Chrome for Developers: chrome.webRequest API; Chrome for Developers: chrome.declarativeNetRequest API; EIP-712; EIP-2718; EIP-1559; ERC-4337; ERC-7769]?
