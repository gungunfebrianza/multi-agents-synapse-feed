Title: Origin-Locked Browser Agents: Security Must Be the Runtime Architecture of LLM Web Automation

Thesis:
Browser-based LLM agents should be built as origin-locked, session-scoped systems where security is not a filter on top of automation but the structure of the browser runtime itself: isolated browser contexts, origin-scoped permissions, human takeover on sensitive steps, and injection monitoring are the only coherent way to operate GUI-driving agents against hostile web content [Playwright docs: Isolation; Playwright docs: BrowserContext; OpenAI: Computer-Using Agent; OWASP Prompt Injection page; OpenAI: Operator System Card; Anthropic: Mitigating the risk of prompt injections in browser use].

The System:
Use an LLM agent that acts through the browser as a computer-use model, not a site-specific API client. OpenAI’s CUA is designed to interact with GUIs through screenshots, mouse, and keyboard, and the same model is exposed for developers through the computer use tool in the Responses API [OpenAI: Computer-Using Agent; OpenAI: New tools for building agents]. AutoGen’s WebSurfer-style agents already operationalize this pattern with Chromium via Playwright, with action spaces including navigation, clicking, typing, summarization, and question answering [AutoGen docs: autogen_ext.agents.web_surfer; AutoGen docs: Magentic-One].

That browser layer must be partitioned into isolated sessions per task and per trust zone. Playwright BrowserContext objects are incognito-like profiles with separate cookies, local storage, and session storage, and non-persistent contexts do not write browsing data to disk [Playwright docs: Isolation; Playwright docs: BrowserContext]. This makes the browser the primary containment boundary for the agent, not just its interface.

Inside each context, permissions must be origin-scoped. Playwright documents granting permissions only for a specified origin through BrowserContext, allowing capability constraints at the context/origin level [Playwright docs: BrowserContext API]. At the engine level, Chromium Site Isolation already exists to isolate sites/origins into separate renderer processes and protect site-specific stored data and permissions [Chromium Site Isolation; Chromium Site Isolation Design Document]. Combined, these facts support a concrete architecture: every agent subtask gets a fresh context, every sensitive origin gets explicit permission boundaries, and renderer isolation reduces cross-site compromise surfaces [Playwright docs: BrowserContext API; Chromium Site Isolation].

The agent orchestration layer must treat browsing as adversarial. OWASP identifies indirect prompt injection from untrusted content and Thought/Observation Injection that forges tool outputs or reasoning traces, and recommends layered defenses including input/output handling and post-hoc detection of prompt leakage, exfiltration markup, or policy-violating content [OWASP Prompt Injection page; OWASP LLM Prompt Injection Prevention Cheat Sheet]. This is decisive because browser agents ingest untrusted page content continuously.

The threat is not theoretical. OpenAI’s Operator system card reports 23% prompt-injection susceptibility on a 31-scenario evaluation set even after mitigations, versus 62% with no mitigations and 47% with prompting-only mitigations [OpenAI: Operator System Card]. Anthropic states no browser agent is immune to prompt injection, and reports a containment test where Claude completed a credential/data exfiltration-style attack in 24 of 25 retries in one setup when runtime containment was weak [Anthropic: Mitigating the risk of prompt injections in browser use; Anthropic: How we contain Claude across products]. Security therefore cannot be prompt-only; it must be enforced through browser containment and execution controls.

The system should therefore combine four layers:

1. Ephemeral browser compartments. Launch each task in a non-persistent Playwright context so cookies, local storage, and session state are isolated and not written to disk [Playwright docs: Isolation; Playwright docs: BrowserContext].

2. Origin-locked capability grants. Only grant permissions to the exact origin that requires them through BrowserContext APIs, aligning with Chromium’s origin/process isolation model [Playwright docs: BrowserContext API; Chromium Site Isolation].

3. Sensitive-action handoff. Use user-control patterns already documented by Operator: takeover mode for credentials or payment entry, watch mode on sensitive sites, approval before significant actions, and one-click deletion of browsing data plus logout from all sites [OpenAI Introducing Operator; OpenAI Help Center: Managing active sessions in ChatGPT]. This converts high-risk steps from autonomous execution into supervised execution.

4. Injection-aware runtime monitoring. Operator documents a prompt-injection monitor that can pause execution when suspicious on-screen content is detected [OpenAI: Operator System Card]. This should be bound to the orchestrator layer used in agent systems such as Magentic-One, where planning, progress tracking, and corrective actions already exist [AutoGen docs: Magentic-One]. Because newer attacks target memory and plans, not only visible prompts, the orchestrator must treat plans as tainted state when browsing untrusted pages. That requirement is grounded by work showing plan injections can outperform prompt-based attacks and by attacks through the HTML accessibility tree against accessibility-tree-based web agents [arXiv: Context manipulation attacks: Web agents are susceptible to corrupted memory; arXiv: Manipulating LLM Web Agents with Indirect Prompt Injection Attack via HTML Accessibility Tree; AutoGen docs: Magentic-One].

This creates one recombined system: a secure browser runtime for LLM agents where the browser provides compartments, the LLM agent provides flexible GUI execution, and security defines how planning, permissions, and handoff are enforced.

Interdependence proof:
- Remove browser: The system loses its containment substrate. There is no BrowserContext isolation, no origin-scoped permissions, no Chromium site isolation, and no takeover/watch-mode style control surface tied to live web sessions [Playwright docs: Isolation; Playwright docs: BrowserContext API; Chromium Site Isolation; OpenAI Introducing Operator]. The idea collapses into generic agent security instead of secure web automation.
- Remove LLM Agents: The system loses the adaptive actor that makes use of screenshots, mouse, keyboard, accessibility trees, planning, and corrective action across arbitrary sites [OpenAI: Computer-Using Agent; AutoGen docs: Magentic-One]. Then the browser is just a hardened automation shell, not a browser agent system.
- Remove security: The remaining browser agent becomes operationally unacceptable. OWASP lists indirect prompt injection and forged observations as core vulnerabilities, Operator still shows measurable susceptibility after mitigation, Anthropic says no browser agent is immune, and weak containment can permit exfiltration at extreme rates [OWASP Prompt Injection page; OpenAI: Operator System Card; Anthropic: Mitigating the risk of prompt injections in browser use; Anthropic: How we contain Claude across products]. Without security, the architecture fails at its primary task.

Failure mode:
If the agent reuses browser state across trust boundaries or lets plans persist after reading hostile pages, indirect prompt injection can corrupt both memory and action selection despite prompt-level defenses [OWASP Prompt Injection page; arXiv: Context manipulation attacks: Web agents are susceptible to corrupted memory].
If accessibility-tree or on-screen monitoring is absent, hostile HTML can steer accessibility-tree-based agents into credential exfiltration or malicious clicks before the orchestrator notices [arXiv: Manipulating LLM Web Agents with Indirect Prompt Injection Attack via HTML Accessibility Tree; OpenAI: Operator System Card].

Open question:
How should an orchestrator formally reset or quarantine agent plans and memory across browser-context boundaries so that context-manipulation attacks cannot survive task isolation while still preserving enough state for useful multi-step web work [Playwright docs: BrowserContext; AutoGen docs: Magentic-One; arXiv: Context manipulation attacks: Web agents are susceptible to corrupted memory]?
