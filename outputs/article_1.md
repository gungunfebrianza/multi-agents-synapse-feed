Title: Consent-Recorded Local Python: A Privacy-by-Design WebAssembly App Pattern

Thesis:
Python should be shipped into the browser through WebAssembly as the default architecture for privacy-sensitive interactive applications, because that combination keeps computation local while preserving Python’s package ecosystem, and privacy then becomes an enforceable product constraint expressed through local processing, minimized filesystem access, retention rules, and explicit consent records for any optional telemetry or identifier collection [Pyodide stable docs – What is Pyodide? (pyodide.org); PyScript docs – Filesystems (docs.pyscript.net); ICO – Data protection by design and by default (ico.org.uk); ICO – How should we obtain, record and manage consent? (ico.org.uk)].

The System:
Build a browser application that runs its core logic in Python on WebAssembly using PyScript with the Pyodide interpreter. This choice is structurally necessary because PyScript is a production framework for running Python in modern browsers on top of WebAssembly, supports Pyodide and MicroPython, and exposes bidirectional FFI with JavaScript and browser APIs through `import js` [PyScript docs – What is PyScript?; Features; Built-in APIs (docs.pyscript.net)]. Use Pyodide, not MicroPython, because Pyodide supports installation of any pure-Python package with a wheel on PyPI via micropip and includes many ported C-extension packages such as NumPy, pandas, SciPy, Matplotlib, and scikit-learn, while PyScript documents that micropip is Pyodide-only and MicroPython does not support PyPI-hosted packages in the same way [Pyodide stable docs – What is Pyodide? (pyodide.org); PyScript docs – Features; Configuration (docs.pyscript.net)].

The transfer move is this: take Python’s established app-shipping logic and retarget it from desktop embedding to browser embedding. The Python Packaging User Guide explicitly organizes shipping by environment, including interpreter-embedded executables and embedded-device deployment, which shows that Python distribution is already framed as “ship the runtime inside the target environment” rather than “require a preinstalled interpreter” [Python Packaging User Guide – Overview of Python Packaging (packaging.python.org)]. The official Windows embeddable distribution is even described as a minimal Python package suitable for embedding into larger applications [Python 3.14.6 docs – Using Python on Windows / embeddable distribution (docs.python.org)]. WebAssembly extends that same packaging mindset into the browser: Pyodide is a Python distribution for the browser and Node.js based on WebAssembly/Emscripten, and its platform is formalized around the `pyemscripten_2025_0` tag with CPython 3.13 and Emscripten 4.0.9 [Pyodide stable docs – What is Pyodide? (pyodide.org); Pyodide docs – pyemscripten_2025_0 platform specification (pyodide.org)].

Privacy is not an add-on policy page in this system. It defines the architecture. The app should process personal data in-browser by default, because ICO guidance requires privacy at the start of everything organizations do, minimization of collected data, deletion of personal information no longer needed through a retention schedule, and easy withdrawal where consent is required [ICO – Data protection by design and by default (ico.org.uk)]. The browser execution model reinforces this: PyScript documents that both Pyodide and MicroPython use an in-browser virtual filesystem by default, and access to user-local filesystems is exposed through `pyscript.fs` only if the user allows it [PyScript docs – Filesystems (docs.pyscript.net)]. That makes local-only execution and user-mediated file access the baseline product shape, not a later security patch.

The privacy layer has four operating rules:

1. **Default local execution**  
   Run analysis, transformation, and inference inside Pyodide in the browser so raw user data stays in the browser session by default [Pyodide stable docs – What is Pyodide? (pyodide.org); PyScript docs – Filesystems (docs.pyscript.net)].

2. **Minimal collection**  
   Do not collect telemetry, identifiers, or uploaded content unless a specific purpose requires it, because ICO guidance requires minimizing collected data [ICO – Data protection by design and by default (ico.org.uk)].

3. **Consent ledger for any optional collection**  
   If the app does collect telemetry or identifiers, record what the person was told, the consent statement or form version, and the date consent was given, because ICO consent guidance requires those records [ICO – How should we obtain, record and manage consent? (ico.org.uk)].

4. **Retention and withdrawal**  
   Delete personal information that is no longer needed according to a retention schedule, and provide an easy way to withdraw consent later [ICO – Data protection by design and by default (ico.org.uk)].

This system also fits current package reality. Use pure-Python wheels where possible, because PyScript’s FAQ states that packages fail when they contain C, C++, or Rust code that has not been compiled to WebAssembly [PyScript FAQ (docs.pyscript.net)]. Where scientific or analytics functionality is required, prefer packages already ported in Pyodide’s ecosystem, since Pyodide already includes many major C-extension packages [Pyodide stable docs – What is Pyodide? (pyodide.org)]. First-load cost must be treated as a packaging budget because PyScript configuration docs confirm that interpreter and package download size materially affect first-load startup, even though package caching reduces delay on later loads [PyScript docs – Configuration (docs.pyscript.net)].

The regulatory scope remains active even with local execution. GDPR still governs collecting, storing, and managing personal data, and special-category data requires explicit consent or another specific legal basis [Your Europe – Data protection under GDPR (europa.eu)]. If the browser app performs on-device AI, the EU AI Act is also in force, with Regulation (EU) 2024/1689 effective since 1 August 2024 and prohibitions effective from February 2025 [European Commission – AI Act / Shaping Europe’s digital future; European Commission health AI page]. Local execution reduces unnecessary transfer, but it does not remove GDPR or AI Act duties.

Interdependence proof:
- Remove python: the system loses the transferable packaging and application logic that makes complex local apps practical. Pyodide’s value is specifically that it is a Python distribution for browser WebAssembly with access to pure-Python wheels and many ported scientific packages; remove Python and the package and runtime strategy described here disappears [Pyodide stable docs – What is Pyodide? (pyodide.org)].
- Remove webassembly: the system loses browser-resident execution. PyScript’s browser runtime, Pyodide’s browser distribution model, the in-browser virtual filesystem default, and user-mediated local file access all depend on Python running on WebAssembly in the browser [PyScript docs – What is PyScript?; Features; Built-in APIs (docs.pyscript.net); PyScript docs – Filesystems (docs.pyscript.net); Pyodide stable docs – What is Pyodide? (pyodide.org)].
- Remove privacy: the architecture collapses into a generic browser Python app with no reason to prefer local-first execution, no minimization rule, no retention schedule, no consent recording requirement, and no fair consent UX constraint. The system’s core design choice—local computation with optional, tightly governed data collection—comes directly from ICO and GDPR requirements [ICO – Data protection by design and by default (ico.org.uk); ICO – How should we obtain, record and manage consent? (ico.org.uk); Your Europe – Data protection under GDPR (europa.eu)].

Failure mode:
If the app depends on Python packages containing C, C++, or Rust code that have not been compiled to WebAssembly, installation fails in the browser and the local-first architecture breaks [PyScript FAQ (docs.pyscript.net)].
If the app adds telemetry or identifiers without recorded consent, retention rules, and non-manipulative consent UX, the privacy claim fails even if all computation is local [ICO – How should we obtain, record and manage consent? (ico.org.uk); ICO – Privacy by design; About this guidance; consent-or-pay materials (ico.org.uk)].

Open question:
How should a browser-native Python app persist auditable consent records and retention actions while preserving the local-first, user-permissioned filesystem model that PyScript exposes through its in-browser virtual filesystem and optional `pyscript.fs` access [PyScript docs – Filesystems (docs.pyscript.net); ICO – How should we obtain, record and manage consent? (ico.org.uk)]?
