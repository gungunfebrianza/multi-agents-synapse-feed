Title: Free-Threaded Python Needs Immutable Authenticated State, Not Faster Mutable Caches

Thesis:
The right way to combine data structure, security, and Python is a Python state system built on immutable authenticated mappings and Merkle proof layers, because Python’s free-threaded runtime raises the cost of shared mutable state, while Python’s serialization ecosystem makes unsafe object reconstruction an explicit code-execution risk, so the core state container must become snapshot-friendly, tamper-evident, and hostile to unsafe deserialization by design [What’s New In Python 3.13 — official docs; PEP 779; Python Free-Threading Guide; Python 3.14 security warnings; Python developer security policy; Python pickle documentation; PyYAML documentation / project guidance; PEP 603 – Adding a frozenmap type to collections].

The System:
Impose the hardest data-structure constraint onto security: make secure state verifiable through persistent authenticated structure rather than through trusted mutable objects.

That constraint is already visible in two families of structures. PEP 603 describes HAMTs as immutable mappings with O(log N) get/set operations, while noting lookup is about 30% slower on average than Python dict lookup [PEP 603 – Adding a frozenmap type to collections]. Ethereum’s authenticated state uses a modified Merkle-Patricia Trie because it is deterministic and cryptographically verifiable, and because a state root plus branch hashes can prove equality or membership-like inclusion for ledger data [ethereum.org: Merkle Patricia Trie documentation]. Pymerkle 6.1.0 shows the practical Python-side pattern: an in-memory Merkle tree can generate inclusion and consistency proofs, and its current proof-generation design is reported as about five times faster while keeping peak memory usage low and sublinear in tree size [pymerkle 6.1.0 documentation (Read the Docs)].

Now impose the hardest security constraint onto Python: the runtime must assume attacker-controlled serialized input is hostile. Python’s own documentation warns that unpickling attacker-controlled data can execute arbitrary code, and the developer security policy explicitly lists pickle, marshal, shelve, eval(), and exec() as mechanisms documented to execute arbitrary Python code supplied as data [Python 3.14 security warnings; Python developer security policy; Python pickle documentation]. PyYAML gives the same direction: yaml.load on untrusted data is unsafe because it can construct arbitrary Python objects, and safe_load is the project guidance for untrusted input [PyYAML documentation / project guidance]. A 2026 fuzzing paper adds that even beyond arbitrary code execution, discrepancies across pickle implementations remain a practical attack surface [arXiv: PickleFuzzer: A Case Study in Fuzzing for Discrepancies Between Python Pickle Implementations (2026)].

What Python must become to satisfy both constraints is a host for authenticated snapshots, not a host for trusted mutable heaps. Python 3.13 provides prebuilt free-threaded binaries and supports building CPython with --disable-gil [What’s New In Python 3.13 — official docs]. PEP 779 and the Free-Threading Guide state that free-threaded Python is no longer considered experimental starting in Python 3.14, even though it is not the default build [PEP 779; Python Free-Threading Guide]. In that environment, shared mutable structures need stronger thread-safety assumptions than under the historical GIL model [What’s New In Python 3.13 — official docs]. That pushes system design toward immutable mappings for concurrent reads and versioned updates, because they reduce dependence on mutation discipline. HAMTs already provide the persistence model [PEP 603 – Adding a frozenmap type to collections], and Merkle structures provide verifiable state roots and proofs [ethereum.org: Merkle Patricia Trie documentation; pymerkle 6.1.0 documentation (Read the Docs)].

So the buildable design is this:
- Store live application state in an immutable HAMT-style mapping for snapshot semantics and O(log N) updates [PEP 603 – Adding a frozenmap type to collections].
- Commit each snapshot to a Merkle layer that emits inclusion and consistency proofs [pymerkle 6.1.0 documentation (Read the Docs)].
- Hash internal nodes with a tree-based hash primitive whose implementation is already optimized for parallelism; the official BLAKE3 implementation describes BLAKE3 as internally Merkle-tree-based and emphasizes high parallelism across threads and SIMD lanes [BLAKE3 official GitHub repository].
- Exchange only plain data plus authenticated proofs, never pickled or unsafe YAML object graphs, because Python and PyYAML both document those object-reconstruction paths as unsafe on untrusted input [Python 3.14 security warnings; Python developer security policy; Python pickle documentation; PyYAML documentation / project guidance].
- Treat caches as derived acceleration only, not as policy-bearing security state. AWS disclosed CVE-2026-6550 in the AWS Encryption SDK for Python, where a shared key cache allowed an authenticated local actor to bypass key-commitment policy enforcement through an algorithm-downgrade path in the caching layer before fixed versions [AWS Security Bulletin 2026-017 (CVE-2026-6550)]. That makes mutable shared caches the wrong place to encode security invariants.
- If Python-side memoization is used, keep it outside trust boundaries; cachetools says its decorators are thread-safe by default, but that statement applies to the decorators and not to arbitrary custom cache mutation patterns [cachetools 7.1.4 documentation]. That is another reason to put correctness in immutable authenticated state rather than in mutable cache coordination.

This system accepts the Python tradeoff directly. HAMT lookup is slower than dict lookup on average [PEP 603 – Adding a frozenmap type to collections]. But Python 3.14 introduces an experimental JIT in official macOS and Windows binaries and bytecodes such as LOAD_FAST_BORROW to reduce reference-counting overhead, both of which can change the runtime cost profile of data-structure-heavy code [CPython 3.14 What’s New draft / main documentation source]. The point is not that Python erases the asymptotic cost. The point is that Python is now changing enough at the runtime level that correctness should be anchored in immutable authenticated structure, while performance tuning can happen around it.

Interdependence proof:
- Remove data structure: You lose the immutable HAMT persistence model and the Merkle proof model, so there is no O(log N) snapshot-friendly map and no deterministic root-plus-branch verification mechanism for state membership or equality claims [PEP 603 – Adding a frozenmap type to collections; ethereum.org: Merkle Patricia Trie documentation]. The system collapses into ordinary mutable Python objects and insecure trust in process memory.
- Remove security: You still have a clever Python container, but nothing prevents developers from moving state through pickle or unsafe yaml.load, both documented as unsafe on untrusted data [Python 3.14 security warnings; Python developer security policy; Python pickle documentation; PyYAML documentation / project guidance]. You also recreate the exact class of “security policy hidden in cache state” failure illustrated by CVE-2026-6550 [AWS Security Bulletin 2026-017 (CVE-2026-6550)].
- Remove python: You lose the reason this design is urgent. Free-threaded CPython changes the concurrency assumptions around shared mutable structures [What’s New In Python 3.13 — official docs; PEP 779; Python Free-Threading Guide]. The proposal is specifically about how Python’s runtime evolution and unsafe serialization warnings force authenticated immutable state to be the native systems pattern.

Failure mode:
If developers add a mutable shared cache that carries authorization or algorithm-selection decisions, they can reintroduce the exact kind of cache-layer policy bypass shown in CVE-2026-6550 [AWS Security Bulletin 2026-017 (CVE-2026-6550)].
If developers serialize snapshots with pickle or unsafe yaml.load, the authenticated state design is bypassed by Python’s own documented arbitrary-code-execution paths [Python 3.14 security warnings; Python pickle documentation; PyYAML documentation / project guidance].

Open question:
Can a Python-native authenticated immutable map combine HAMT-style O(log N) persistence with Merkle proof generation near pymerkle’s current proof-efficiency profile, while exploiting free-threaded execution and BLAKE3’s parallel tree hashing without making proof sizes and proving times drift toward the large, seconds-scale costs reported for current Verkle benchmarks [PEP 603 – Adding a frozenmap type to collections; pymerkle 6.1.0 documentation (Read the Docs); BLAKE3 official GitHub repository; arXiv: Towards Stateless Clients in Ethereum: Benchmarking Verkle Trees and Binary Merkle Trees with SNARKs (2025)]?
