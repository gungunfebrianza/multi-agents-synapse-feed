Title: Origin-Scoped Permit Firewall: a Browser Extension That Turns Smart-Contract Signing Into a Security Gate

Thesis:
The right buildable Web3 defense is a browser extension that requests host access only at the moment a user enters a dapp, decodes smart-contract signatures and transactions with special handling for EIP-712 and ERC-2612 permit flows, and sends them to pre-sign simulation plus reputation systems before the wallet action completes, because modern attacks bypass static whitelists and broad browser interception is no longer a viable extension architecture [MDN: optional_host_permissions (developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/optional_host_permissions)] [Ethereum Improvement Proposals: EIP-712 (eips.ethereum.org/EIPS/eip-712)] [Ethereum Improvement Proposals: ERC-2612 (eips.ethereum.org/EIPS/eip-2612)] [Blockaid Blog: False sense of security: the whitelist security gaps your wallet doesn’t tell you about (blockaid.io/blog/false-sense-of-security-the-whitelist-security-gaps-your-wallet-doesnt-tell-you-about)] [Chrome for Developers: chrome.webRequest API (developer.chrome.com/docs/extensions/reference/api/webRequest); Chrome for Developers: chrome.declarativeNetRequest API (developer.chrome.com/extensions/declarativeNetRequest)].

The System:
The system is an MV3 browser extension designed around least privilege and user-triggered analysis rather than full network control.

1. Permission model.
In Chrome, programmatic injection requires either "activeTab" or host permissions, and changing host match patterns produces user-facing permission warnings, while Chrome recommends optional permissions to reduce consent friction [Chrome for Developers: Declare permissions (developer.chrome.com/docs/extensions/develop/concepts/declare-permissions)]. In Firefox, "optional_host_permissions" can be granted at runtime and later revoked by the user [MDN: optional_host_permissions (developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/optional_host_permissions)].  
So the extension requests origin access only when the user visits a dapp and explicitly asks for a risk check. That makes marketplace compliance part of the design, because Chrome reviews broad-access extensions for abuse and Mozilla policy requires secure handling and prohibits abusive crypto behavior [Chrome Web Store review process (developer.chrome.com/docs/webstore/review-process); Firefox Extension Workshop: Add-on Policies (extensionworkshop.com/documentation/publish/add-on-policies/)].

2. Execution model.
The extension does not depend on imperatively blocking wallet or RPC traffic. In Chrome MV3, ordinary extensions cannot use "webRequestBlocking"; blocking and modification generally must use declarativeNetRequest [Chrome for Developers: chrome.webRequest API (developer.chrome.com/docs/extensions/reference/api/webRequest); Chrome for Developers: chrome.declarativeNetRequest API (developer.chrome.com/extensions/declarativeNetRequest)]. In Firefox, content scripts cannot use host permissions to perform cross-origin fetches; those calls must move to extension pages or background logic [MDN: Content scripts (developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/Content_scripts)].  
So the architecture is: content script reads page and wallet-request context, then sends normalized signing data to the extension background or extension page, which performs RPC, simulation, and reputation queries.

3. Smart-contract decoding core.
The extension treats smart-contract semantics as the primary unit of analysis. EIP-712 defines typed structured signing over "\x19\x01" || domainSeparator || hashStruct(message), which allows wallets and tools to decode fields and present human-readable signals [Ethereum Improvement Proposals: EIP-712 (eips.ethereum.org/EIPS/eip-712)]. ERC-2612 standardizes permit-based signed approvals that change allowances without a direct on-chain approve transaction from msg.sender [Ethereum Improvement Proposals: ERC-2612 (eips.ethereum.org/EIPS/eip-2612)].  
That means the extension must not merely label “signature request.” It must classify whether the signature is a permit-style approval, identify spender and token fields, and mark unlimited-approval risk, because permit phishing is a known bypass path for wallet and policy whitelists [Blockaid Blog: False sense of security: the whitelist security gaps your wallet doesn’t tell you about (blockaid.io/blog/false-sense-of-security-the-whitelist-security-gaps-your-wallet-doesnt-tell-you-about)].

4. Security engine.
The extension sends supported transactions and signature requests for pre-sign analysis, following the same pattern used by production wallets. MetaMask states its security alerts send transactions and signature requests to simulation providers before execution, combining phishing-domain intelligence, contract behavior, impersonation signals, malicious-campaign activity, and community reporting [MetaMask Help Center: How to manage security alerts to protect your wallet (support.metamask.io/configure/wallet/security-alerts/)]. MetaMask also identifies malicious token approvals as a common attack and says supported transactions are analyzed before being sent to the dapp [MetaMask Help Center: Transaction security checks (support.metamask.io/stay-safe/safety-in-web3/using-transaction-security-providers-to-protect-yourself-from-scams)].  
For implementation, the extension can call mature off-chain simulation systems. Tenderly’s production case studies show transaction simulation against mainnet data before gas is spent and exposure through major app workflows [Tenderly Safe case study PDF (tenderly.co/pdf/Tenderly-Safe-Case-Study.pdf); Tenderly Instadapp case study PDF (tenderly.co/pdf/Tenderly-Instadapp-Case-Study.pdf)]. Blockaid advertises simulation and validation across Ethereum mainnet and multiple EVM chains, with Coinbase Wallet using it for safety and UX [Blockaid: Transaction Simulation and Validation Engine (blockaid.io/transaction-security)].

5. Address-poisoning defense layer.
The extension also checks destination addresses and recent-history lookalikes before the signature prompt. Address poisoning is not edge-case noise: a USENIX Security 2025 paper measured 270 million attack attempts targeting 17 million victims and verified 6,633 incidents with at least $83.8 million in losses between January 2022 and June 30, 2024 [USENIX Security 2025 paper: Blockchain Address Poisoning (usenix.org/system/files/usenixsecurity25-tsuchiya.pdf)]. Chainalysis reported that in 2024 crypto sent to address-poisoning scams grew by more than 15,000% [Chainalysis 2025 Crypto Crime Report release PDF (chainalysis.com/wp-content/uploads/2025/03/the-2025-crypto-crime-report-release.pdf)].  
So the extension flags copied or recently seen lookalike destinations and forces explicit confirmation when a transaction target resembles poisoned history.

6. Why this recombination is adversarially correct.
Browser extension constraints force the system away from total interception and toward user-initiated, origin-scoped analysis [Chrome for Developers: chrome.webRequest API (developer.chrome.com/docs/extensions/reference/api/webRequest); Chrome for Developers: chrome.declarativeNetRequest API (developer.chrome.com/extensions/declarativeNetRequest)] [Chrome for Developers: Declare permissions (developer.chrome.com/docs/extensions/develop/concepts/declare-permissions)].
Security reality forces the system away from static whitelists and toward simulation plus reputation [MetaMask Help Center: How to manage security alerts to protect your wallet (support.metamask.io/configure/wallet/security-alerts/)] [Blockaid Blog: False sense of security: the whitelist security gaps your wallet doesn’t tell you about (blockaid.io/blog/false-sense-of-security-the-whitelist-security-gaps-your-wallet-doesnt-tell-you-about)].
Smart-contract structure forces the system to decode EIP-712 and ERC-2612 semantics rather than treat all signatures as generic prompts [Ethereum Improvement Proposals: EIP-712 (eips.ethereum.org/EIPS/eip-712)] [Ethereum Improvement Proposals: ERC-2612 (eips.ethereum.org/EIPS/eip-2612)].

Interdependence proof:
- Remove browser extension: The defense loses the only component positioned at the dapp entry point, where origin-specific permission requests, page-context inspection, and user-triggered analysis can occur under MV3 rules [Chrome for Developers: Declare permissions (developer.chrome.com/docs/extensions/develop/concepts/declare-permissions)] [MDN: optional_host_permissions (developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/optional_host_permissions)]. The system becomes just another backend scanner with no origin-aware intervention surface.
- Remove security: The extension becomes a decoder without protection. Static display of contract fields fails against the exact attack pattern already documented: typed-message phishing and malicious permit flows bypassing whitelists, which production systems address with simulation, phishing intelligence, impersonation signals, and policy enforcement [Blockaid Blog: False sense of security: the whitelist security gaps your wallet doesn’t tell you about (blockaid.io/blog/false-sense-of-security-the-whitelist-security-gaps-your-wallet-doesnt-tell-you-about)] [MetaMask Help Center: How to manage security alerts to protect your wallet (support.metamask.io/configure/wallet/security-alerts/)].
- Remove smart contract: The extension cannot distinguish harmless signing from allowance transfer risk. EIP-712 is what makes typed fields decodable, and ERC-2612 is a major phishing surface that specifically requires permit-aware classification [Ethereum Improvement Proposals: EIP-712 (eips.ethereum.org/EIPS/eip-712)] [Ethereum Improvement Proposals: ERC-2612 (eips.ethereum.org/EIPS/eip-2612)]. Without smart-contract semantics, the system misses the critical difference between a login signature and an unlimited approval.

Failure mode:
If the extension relies on domain trust or wallet whitelists alone, typed-message phishing and malicious permit flows bypass it [Blockaid Blog: False sense of security: the whitelist security gaps your wallet doesn’t tell you about (blockaid.io/blog/false-sense-of-security-the-whitelist-security-gaps-your-wallet-doesnt-tell-you-about)].
If it relies on broad network interception, MV3 platform limits and store-policy pressure break deployment before attackers do [Chrome for Developers: chrome.webRequest API (developer.chrome.com/docs/extensions/reference/api/webRequest); Chrome for Developers: chrome.declarativeNetRequest API (developer.chrome.com/extensions/declarativeNetRequest)] [Chrome Web Store review process (developer.chrome.com/docs/webstore/review-process); Firefox Extension Workshop: Add-on Policies (extensionworkshop.com/documentation/publish/add-on-policies/)].

Open question:
How should a browser extension fuse EIP-712 field decoding, permit-specific spender analysis, simulation results, phishing-domain intelligence, and address-poisoning lookalike detection into one pre-sign decision that is accurate enough to stop attacks without overwhelming users with warnings?
