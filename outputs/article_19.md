Title: Extension-Enforced Contract Intent Firewall

Thesis:
The right recombination of browser extension, security, and smart contract is a contract-intent firewall: a browser extension that intercepts wallet-facing user journeys, verifies whether a smart contract is readable and attributable, simulates the exact contract effect before signing, and blocks or escalates risky approval and signature patterns. This system is necessary because the live attack surface already combines extension impersonation, phishing overlays, malicious signing requests, and opaque contract interactions across the browser boundary [Malwarebytes: Fake Pudgy World site steals your crypto passwords; Ethereum Improvement Proposal EIP-712; MetaMask Help Center: What are estimated balance changes?].

The System:
A contract-intent firewall is a browser extension whose security model is built around the fact that the browser is where users encounter malicious ads, phishing pages, fake wallet prompts, and signing requests. Google’s 2025 scam advisory explicitly warned that malvertising targets sophisticated users with crypto wallets and can serve as the first step in a broader abuse chain [Google Blog: Trust & Safety scam advisory (May 2025)]. Malwarebytes then documented a concrete phishing pattern: a fake Pudgy World site displayed overlays imitating real wallet browser-extension unlock screens, including wallet-specific logos, colors, and wording, to steal credentials [Malwarebytes: Fake Pudgy World site steals your crypto passwords]. That makes the extension the correct enforcement point.

The extension’s first layer is page-side observation and warning. This layer is constrained by actual extension platform rules: Chrome Manifest V3 no longer gives most extensions webRequestBlocking, so an extension can observe requests with webRequest but generally must use Declarative Net Request for blocking or modification behavior [Chrome for Developers: chrome.webRequest API]. Firefox content scripts require host permissions and do not run on restricted domains, which means enforcement coverage must be permission-scoped rather than assumed universal [MDN Web Docs: Content scripts]. Those platform constraints force the system to separate detection from blocking and to design around partial visibility.

The second layer is smart-contract readability. The extension should resolve whether the destination contract is verified, obtain a verified ABI when available, and map proxies to implementations before presenting action summaries. This is grounded in current tooling: Etherscan exposes ABI retrieval for verified contracts, and its verification flow exists to match source to on-chain bytecode for transparency [Etherscan API: Sample API Calls; Etherscan: Verify & Publish Contract Source Code]. Etherscan also now exposes a verifyproxycontract path with chainid support and optional expectedimplementation, which helps map proxy addresses to readable implementation contracts [Etherscan Docs: Verify Proxy Contract]. Without that resolution step, a user signs against bytecode opacity.

The third layer is pre-signing contract simulation. MetaMask’s current human-readable pre-signing insight is transaction simulation with estimated balance changes for contract interactions on Ethereum, including predicted ERC-20, ERC-721, and ERC-1155 assets moving in and out of the account before submission [MetaMask Help Center: What are estimated balance changes?]. A contract-intent firewall should use that same category of insight as its core decision surface: before a user signs, show expected asset deltas and compare them with the page’s claimed intent. If the page says “claim reward” and the simulation shows outbound NFTs or token approvals, the extension raises a hard warning. This is already validated as a defensive pattern in the ecosystem: wallet-adjacent extensions advertise transaction simulation, phishing warnings, and scam detection before signing [Microsoft Edge Add-ons: Web3 Antivirus].

The fourth layer is signature-specific phishing defense. EIP-712 explicitly states that user agents may do contract-specific phishing prevention [Ethereum Improvement Proposal EIP-712]. That means typed structured data signing is not just a serialization format; it is a security surface the extension should actively interpret. The extension should classify whether a typed-data request is granting spending power, delegating authority, or confirming benign app state, then require different UI treatment for each class. This is especially important for allowance minimization flows such as Permit2, whose feature set includes expiring approvals, one-time signature-based transfers, batch approvals/transfers, and batch revocation to reduce the risk of long-lived token allowances [Uniswap Blog: How to Integrate with Permit2]. A firewall can use those semantics to distinguish safer bounded approvals from broad persistent risk.

The fifth layer is privacy-aware simulation routing. MetaMask states that estimated balance changes rely on a centralized MetaMask server for simulation and that users who do not want transaction data or IP address exposed should keep the feature disabled [MetaMask Help Center: What are estimated balance changes?]. So the system must expose simulation as a security control with an explicit privacy tradeoff, not as an invisible backend assumption. Security here is not only “detect more”; it is “make the inspection path itself visible.”

The sixth layer is extension supply-chain hardening. Browser stores already require self-contained, reviewable extension code and prohibit remote code execution from outside the package [Chrome for Developers: Chrome Web Store Program Policies; Chrome Extensions: Deal with remote hosted code violations; Firefox Extension Workshop: Add-on Policies]. That matters directly because Microsoft documented a 2025 software supply-chain campaign that compromised hundreds of npm packages and used preinstall execution to run malicious code before tests or security checks, targeting developer environments and CI/CD for credential theft [Microsoft Security Blog: Shai-Hulud 2.0]. So the contract-intent firewall must be architected as a self-contained extension with reviewable functionality, minimizing dependency risk and eliminating remotely hosted executable logic. In this design, browser-extension policy becomes part of the security architecture, not an afterthought.

Interdependence proof:
- Remove browser extension: the system loses the only component positioned inside the user’s live browsing session where malvertising, phishing pages, and fake wallet overlays appear, so it cannot detect or mediate the attack chain at the point of user interaction [Google Blog: Trust & Safety scam advisory (May 2025); Malwarebytes: Fake Pudgy World site steals your crypto passwords].
- Remove security: the system collapses into a passive contract viewer. ABI lookup, proxy resolution, typed-data interpretation, and simulation no longer produce enforcement, warnings, or phishing resistance, despite EIP-712 explicitly making user-agent phishing prevention part of the signing surface [Ethereum Improvement Proposal EIP-712; Etherscan API: Sample API Calls; Etherscan Docs: Verify Proxy Contract; MetaMask Help Center: What are estimated balance changes?].
- Remove smart contract: the extension becomes a generic anti-phishing tool and loses its core ability to explain contract interactions, map proxies, interpret approvals, simulate asset effects, and reason about allowance structures like Permit2 [Etherscan: Verify & Publish Contract Source Code; Etherscan Docs: Verify Proxy Contract; Uniswap Blog: How to Integrate with Permit2; MetaMask Help Center: What are estimated balance changes?].

Failure mode:
A fake site can mimic wallet-extension UI inside the page before the real extension flow starts, so page-level trust signals alone fail [Malwarebytes: Fake Pudgy World site steals your crypto passwords].  
A simulation layer can improve pre-signing safety but may expose transaction data and IP address when routed through a centralized server, so the inspection path itself becomes a security boundary [MetaMask Help Center: What are estimated balance changes?].

Open question:
How should a browser extension combine verified ABI retrieval, proxy-to-implementation resolution, EIP-712 phishing prevention, and transaction simulation into one pre-signing verdict without relying on centralized simulation infrastructure [Etherscan API: Sample API Calls; Etherscan Docs: Verify Proxy Contract; Ethereum Improvement Proposal EIP-712; MetaMask Help Center: What are estimated balance changes?]?
