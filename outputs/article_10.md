Title: Mutable-by-Construction Indexes Need Incremental Algorithms, Not Rebuilds

Thesis:
The right way to combine data structure, mutable, and algorithm is a mutable index that treats every structural change as an incremental algorithmic update against a write-optimized data structure, because the strongest results in the provided facts come from systems that avoid full rebuilds, minimize mutation cost, and preserve correctness through explicit update machinery rather than static recomputation [Write-Optimized and High-Performance Hashing Index Scheme for Persistent Memory, USENIX OSDI 2018; DGAP: Efficient Dynamic Graph Analysis on Persistent Memory, arXiv; Incremental Offline/Online PIR, USENIX Security 2022; Dynamic Graph Algorithms with Batch Updates in the Massively Parallel Computation Model, arXiv].

The System:
Start from the hardest constraint in mutable: mutations must stay safe, cheap, and recoverable under concurrency and persistent state. Rust’s ownership, borrowing, and type checking move many concurrency errors to compile time and prevent data races before execution [Fearless Concurrency - The Rust Programming Language, official Rust documentation]. DGAP shows that mutable persistent state performs well only when the representation is organized around reduced write amplification and crash-consistent repair, using a single mutable CSR, per-section edge logs, and per-thread undo logs [DGAP: Efficient Dynamic Graph Analysis on Persistent Memory, arXiv]. Occualizer shows update paths are where synchronization should be injected, while traversals can remain unsynchronized [Occualizer: Optimistic Concurrent Search Trees From Sequential Code, USENIX OSDI 2022].

Impose that mutable constraint onto data structure. The data structure cannot be a rebuild-heavy static layout. It must support dynamic operations, high load, and low-write resizing. Level Hashing provides exactly that shape: a sharing-based two-level hash table with worst-case constant-scale search, insertion, deletion, and update, while making resizing cost-efficient and extra NVM writes rare [Write-Optimized and High-Performance Hashing Index Scheme for Persistent Memory, USENIX OSDI 2018]. Dash reinforces the target envelope for mutable persistent indexing by supporting dynamic, high-update workloads, load factors above 90%, and recovery in 57 ms regardless of data size [Dash: Scalable Hashing on Persistent Memory, arXiv]. If the workload is massively parallel, Hive Hash Table proves that a dynamically resizable structure can sustain mixed insert-delete-lookup workloads at very high throughput and up to 95% load factor [Hive Hash Table: A Warp-Cooperative, Dynamically Resizable Hash Table for GPUs, arXiv].

Now algorithm must become incremental update orchestration over mutable structure state. The algorithm cannot recompute global structure after each change, because the facts consistently favor incremental maintenance over rebuilds. Incremental Offline/Online PIR updates preprocessing state proportionally to the number of changes and can make updates up to 56x cheaper than rebuilding from scratch [Incremental Offline/Online PIR, USENIX Security 2022]. Dynamic graph algorithms in MPC maintain mutable state so updates need fewer rounds than static recomputation, and can process update batches of size Θ(S) for some problems in O(1) rounds [Dynamic Graph Algorithms with Batch Updates in the Massively Parallel Computation Model, arXiv]. Riker is explicitly positioned as a correctness-guaranteed alternative to rebuild-heavy workflows [Riker: Always-Correct and Fast Incremental Builds from Simple Specifications, USENIX ATC 2022]. Incremental linking achieves in-place updates by preserving relocation and symbol information plus padding, making large-program links on average ten times faster than normal linking [Incremental Linking on HP-UX, USENIX].

That yields one buildable idea: a mutable index engine where the data structure is a write-optimized, dynamically resizable persistent table, and the algorithm is an incremental maintenance planner that updates only affected structural summaries, logs, and synchronization points after each mutation. The planner borrows from incremental preprocessing and batch-update algorithms: it updates metadata in proportion to changed regions instead of full index size [Incremental Offline/Online PIR, USENIX Security 2022], and groups mutations into batches when maintained state allows lower-latency updates than full recomputation [Dynamic Graph Algorithms with Batch Updates in the Massively Parallel Computation Model, arXiv]. The mutable representation borrows DGAP’s principle of localized edge logs and undo logs for crash consistency [DGAP: Efficient Dynamic Graph Analysis on Persistent Memory, arXiv], while the core table borrows Level Hashing’s low-write resize behavior and constant-scale update/search properties [Write-Optimized and High-Performance Hashing Index Scheme for Persistent Memory, USENIX OSDI 2018]. Concurrency control follows Occualizer’s split: synchronize update steps, keep read traversals lean [Occualizer: Optimistic Concurrent Search Trees From Sequential Code, USENIX OSDI 2022]. Safety boundaries follow Rust’s compile-time mutability discipline [Fearless Concurrency - The Rust Programming Language, official Rust documentation].

This system directly resolves the constraint collision. Data structures want stable invariants. Mutable workloads destroy stability through insertions, deletions, and resizes. The algorithm absorbs that collision by becoming an always-on incremental maintainer of invariants, not a periodic rebuilder. Tangram’s motivation supports the core operational reason: mutable abstractions are more efficient for iterative workloads because they permit in-place state updates and asynchronous execution [Tangram: Bridging Immutable and Mutable Abstractions for Distributed Data Analytics, USENIX ATC 2019].

Interdependence proof:
- Remove data structure: The idea collapses into a generic update policy with nowhere to anchor low-write resizing, constant-scale updates, recovery behavior, or high-load operation. The system loses the concrete mechanisms shown by Level Hashing, Dash, and Hive Hash Table [Write-Optimized and High-Performance Hashing Index Scheme for Persistent Memory, USENIX OSDI 2018; Dash: Scalable Hashing on Persistent Memory, arXiv; Hive Hash Table: A Warp-Cooperative, Dynamically Resizable Hash Table for GPUs, arXiv].
- Remove mutable: The need for incremental maintenance largely disappears. Static rebuilds become acceptable, and the advantages proven by DGAP, Tangram, and Incremental Offline/Online PIR no longer drive the design [DGAP: Efficient Dynamic Graph Analysis on Persistent Memory, arXiv; Tangram: Bridging Immutable and Mutable Abstractions for Distributed Data Analytics, USENIX ATC 2019; Incremental Offline/Online PIR, USENIX Security 2022].
- Remove algorithm: The system degenerates into a mutable container without a disciplined way to preserve invariants cheaply after change. It loses the demonstrated gains of maintained state, proportional update work, and correctness-guaranteed incremental repair [Dynamic Graph Algorithms with Batch Updates in the Massively Parallel Computation Model, arXiv; Incremental Offline/Online PIR, USENIX Security 2022; Riker: Always-Correct and Fast Incremental Builds from Simple Specifications, USENIX ATC 2022].

Failure mode:
If mutations trigger whole-structure retraining, rebuilding, or resynchronization, the design violates the incremental principle and forfeits the update-cost advantages shown by Incremental Offline/Online PIR and dynamic batch-update algorithms [Incremental Offline/Online PIR, USENIX Security 2022; Dynamic Graph Algorithms with Batch Updates in the Massively Parallel Computation Model, arXiv].
If update paths are not isolated with explicit synchronization and recovery logs, concurrency safety and crash consistency fail against the constraints demonstrated by Rust, Occualizer, and DGAP [Fearless Concurrency - The Rust Programming Language, official Rust documentation; Occualizer: Optimistic Concurrent Search Trees From Sequential Code, USENIX OSDI 2022; DGAP: Efficient Dynamic Graph Analysis on Persistent Memory, arXiv].

Open question:
How far can a write-optimized mutable data structure push proportional-to-change incremental maintenance before its synchronization, logging, and resize metadata outweigh the benefits of avoiding full recomputation?
