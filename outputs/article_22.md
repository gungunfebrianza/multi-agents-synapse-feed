Title: Capability-Gated Transaction Firewall: a Browser Extension That Enforces Smart-Contract Security Before Wallet Approval

Thesis:
The strongest buildable defense for browser-based crypto flows is a browser extension that uses opt-in tab access to inspect dapp transaction requests, applies origin and scam-risk security signals, and blocks wallet approval unless the target smart contract call survives preflight simulation and contract-verification checks; remove any one of the three parts and the defense collapses.

The System:
Build a Manifest V3 browser extension that acts as a transaction firewall at the exact seam where dapps, wallets, and users meet.

At the browser-extension layer, the system uses MV3’s activeTab permission so inspection only activates after a user gesture on the current tab, giving temporary host access, script injection ability, and request interception ability for that tab’s main-frame origin [Chrome for Developers: https://developer.chrome.com/docs/extensions/develop/concepts/activeTab]. This matters because MV3 removed persistent background pages in favor of an extension service worker and removed blocking webRequest for most extensions, pushing request control toward declarativeNetRequest [Chrome for Developers: https://developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3 and https://developer.chrome.com/docs/extensions/reference/api/webRequest]. The extension therefore uses service-worker orchestration plus declarativeNetRequest rules for host-scoped blocking, redirect, or modification without exposing request contents to the extension, while still requiring host permissions before actions apply [Chrome for Developers: https://developer.chrome.com/docs/extensions/reference/api/declarativeNetRequest]. On Firefox, the same design maps to user-grantable host permissions and activeTab-based programmatic injection, which preserves explicit user consent for inspection [MDN Web Docs: https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/manifest.json/host_permissions and https://developer.mozilla.org/en-US/docs/Mozilla/Add-ons/WebExtensions/Content_scripts].

At the security layer, the extension is designed so its own attack surface is constrained by platform rules. MV3 and Chrome Web Store policy prohibit remotely hosted executable code, including remote JavaScript and WASM, so all executable logic for transaction inspection must be bundled inside the extension package [Chrome for Developers: https://developer.chrome.com/docs/extensions/develop/migrate/what-is-mv3 and https://developer.chrome.com/docs/extensions/develop/migrate/remote-hosted-code]. That directly reduces post-review code-swapping risk. Chrome Web Store also requires 2-Step Verification for developer accounts before publishing or updating an extension, which adds account-takeover resistance for malicious-update scenarios [Chrome for Developers: https://developer.chrome.com/docs/webstore/program-policies/policies]. For live risk signals, the system consumes domain validation and scam indicators equivalent to WalletConnect Verify states—VALID, INVALID, UNKNOWN, plus isScam—to warn or hard-stop suspicious origin connections [WalletConnect Docs: https://docs.walletconnect.network/wallet-sdk/web/verify]. It also mirrors wallet-grade transaction screening by incorporating domain, address, token, and transaction risk signals of the kind MetaMask documents in its security alerts [MetaMask Help Center: https://support.metamask.io/configure/wallet/security-alerts/]. As a browser-native outer layer, it benefits from Chrome Safe Browsing protections against phishing, malware, malicious extensions, abusive sites, and social-engineering attacks [Google Support: https://support.google.com/chrome/a/answer/13616206 and https://support.google.com/chrome/answer/9890866].

At the smart-contract layer, the extension does not trust the dapp’s description of what a signature means. It independently preflights contract execution with eth_call and forecasts gas with eth_estimateGas, both of which execute locally on a node without submitting a transaction [ethereum.org JSON-RPC API: https://ethereum.org/developers/docs/apis/json-rpc/ and EIP-1474: https://eips.ethereum.org/EIPS/eip-1474]. For target understanding, it pulls verified ABI and source metadata through Etherscan’s Get Contract Source Code endpoint and checks verification status, using the fact that Etherscan verification matches uploaded source against deployed on-chain bytecode [Etherscan Docs: https://docs.etherscan.io/api-reference/endpoint/getsourcecode and https://docs.etherscan.io/api-reference/endpoint/checkverifystatus; https://docs.etherscan.io/contract-verification/whats-contract-verification and https://etherscan.io/verifyContract]. For advanced dapps, it also reads wallet capabilities defined by EIP-5792 and can present or gate multi-call flows around wallet_getCapabilities, wallet_sendCalls, wallet_getCallsStatus, and wallet_showCallsStatus [EIP-5792: https://eips.ethereum.org/EIPS/eip-5792]. The result is simple: if origin risk is bad, if the contract is opaque, or if preflight predicts harmful behavior, the extension blocks progression to wallet approval.

This is adversarial recombination because each keyword attacks the others’ weakest assumption. Browser extensions are dangerous if overprivileged, so activeTab and host-scoped controls constrain them [Chrome for Developers: https://developer.chrome.com/docs/extensions/develop/concepts/activeTab; https://developer.chrome.com/docs/extensions/reference/api/declarativeNetRequest]. Security products are weak if they only score domains, so smart-contract simulation and verification force them to prove transaction intent [ethereum.org JSON-RPC API: https://ethereum.org/developers/docs/apis/json-rpc/; Etherscan Docs: https://docs.etherscan.io/contract-verification/whats-contract-verification]. Smart-contract safety is weak if it begins only inside the wallet, so the browser extension moves enforcement one step earlier, at the webpage origin where phishing and spoofed prompts begin [Google Support: https://support.google.com/chrome/answer/9890866; WalletConnect Docs: https://docs.walletconnect.network/wallet-sdk/web/verify].

Interdependence proof:
- Remove browser extension: The system loses the browser-resident enforcement point that can activate on the current tab after a user gesture, inject inspection logic, and apply host-scoped request controls before the wallet flow proceeds [Chrome for Developers: https://developer.chrome.com/docs/extensions/develop/concepts/activeTab and https://developer.chrome.com/docs/extensions/reference/api/declarativeNetRequest]. Then the design becomes only a wallet-side or server-side checker, which no longer controls the dapp-origin attack surface.
- Remove security: The system becomes a raw transaction decoder with no defense against phishing origins, scam domains, malicious update risk, or extension supply-chain abuse. Without bundled-code rules, 2-Step Verification, Safe Browsing coverage, and domain/scam risk signals, the inspector itself and the pages it inspects become untrusted [Chrome for Developers: https://developer.chrome.com/docs/extensions/develop/migrate/remote-hosted-code; https://developer.chrome.com/docs/webstore/program-policies/policies; Google Support: https://support.google.com/chrome/answer/9890866; WalletConnect Docs: https://docs.walletconnect.network/wallet-sdk/web/verify].
- Remove smart contract: The system can score origins but cannot prove what a requested approval will do on-chain. Without eth_call, eth_estimateGas, verified ABI/source retrieval, verification status, and wallet call-capability handling, it cannot distinguish benign requests from dangerous contract interactions before signature [ethereum.org JSON-RPC API: https://ethereum.org/developers/docs/apis/json-rpc/; Etherscan Docs: https://docs.etherscan.io/api-reference/endpoint/getsourcecode and https://docs.etherscan.io/contract-verification/whats-contract-verification; EIP-5792: https://eips.ethereum.org/EIPS/eip-5792].

Failure mode:
A malicious site can present a transaction from a clean-looking domain while targeting an unverified contract whose real behavior only becomes obvious without ABI/source and preflight checks; this system stops that by requiring both origin-risk clearance and smart-contract simulation [WalletConnect Docs: https://docs.walletconnect.network/wallet-sdk/web/verify; Etherscan Docs: https://docs.etherscan.io/contract-verification/whats-contract-verification; ethereum.org JSON-RPC API: https://ethereum.org/developers/docs/apis/json-rpc/].
A malicious extension update can turn an inspector into spyware; this system reduces that path by relying on bundled executable code only and a store publishing flow protected by 2-Step Verification [Chrome for Developers: https://developer.chrome.com/docs/extensions/develop/migrate/remote-hosted-code and https://developer.chrome.com/docs/webstore/program-policies/policies].

Open question:
How should a browser extension normalize and enforce one user-visible risk policy across single-call simulation via eth_call and multi-call wallet-native flows defined by EIP-5792 without losing the origin-validation signals supplied during connection and request time [ethereum.org JSON-RPC API: https://ethereum.org/developers/docs/apis/json-rpc/; EIP-5792: https://eips.ethereum.org/EIPS/eip-5792; WalletConnect Docs: https://docs.walletconnect.network/wallet-sdk/web/verify]?
