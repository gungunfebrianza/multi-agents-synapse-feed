Title: Capability-Gated Browser Wallet Firewall

Thesis:
The right way to secure browser-based smart-contract signing is to turn the browser extension into a capability-gated transaction firewall that only injects on demand, only exposes smart-contract paths the wallet can prove it supports, and converts risky raw approvals into human-readable, policy-checked flows using typed signing, bundled calls, and sponsored execution [Chrome for Developers: "The activeTab permission"; Chrome for Developers: "Content scripts"; Chrome for Developers: "Message passing"; EIP-5792 official specification; EIP-7677 official specification; EIP-2612 official specification; EIP-712 official specification; MetaMask Docs: "Sign data"; Ledger Developer Portal: "Clear Signing for dApps – Get started"].

The System:
Build a wallet extension that never behaves like a permanently injected signer. In Chrome Manifest V3, background logic must live in an extension service worker that runs only when needed, and executable JavaScript must be packaged with the extension rather than remotely hosted [Chrome for Developers: "Extensions / Manifest V3"; Chrome for Developers: "Migrate to a service worker"]. That constraint should define the architecture: ephemeral privileged logic, no remote hot-patching, and persisted policy state outside long-lived background memory [Chrome for Developers: "Extensions / Manifest V3"; Chrome for Developers: "Migrate to a service worker"].

The extension injects only with activeTab or explicit host permission, instead of broad default page presence [Chrome for Developers: "Declare permissions"; Chrome for Developers: "The activeTab permission"]. Its content script runs in Chrome’s isolated world, sharing the DOM but not JavaScript variables with the page, and all privileged actions cross a message-passing boundary through runtime.sendMessage() or runtime.connect() [Chrome for Developers: "Content scripts"; Chrome for Developers: "Message passing"]. That boundary becomes the security choke point.

At that choke point, the extension first calls wallet_getCapabilities so the dapp-facing side can discover exactly which advanced wallet features are supported before presenting a signing route [EIP-5792 official specification; Base Docs: "wallet_getCapabilities"]. If the wallet advertises bundled-call support, the extension routes actions through wallet_sendCalls rather than prompting users with fragmented transaction sequences [EIP-5792 official specification; Base Docs: "wallet_sendCalls"]. If the wallet advertises paymaster capability, the extension uses the EIP-7677 path so sponsored execution and policy-mediated call construction are tied to declared wallet support instead of ad hoc gas UX [EIP-7677 official specification].

The smart-contract layer is the second firewall. For token approvals, the extension prefers ERC-2612 permit, which replaces a separate on-chain approve transaction with a signature flow specified around EIP-712 structured data [EIP-2612 official specification; EIP-712 official specification]. For other actions, the extension prefers EIP-712 signing because MetaMask documents eth_signTypedData_v4 as its most human-readable signing method [MetaMask Docs: "Sign data"]. On compatible hardware-wallet paths, the extension surfaces clear-signing metadata and transaction simulation signals because Ledger documents human-readable clear-signing and a Transaction Check that can simulate EVM transactions and flag likely risks before approval [Ledger Developer Portal: "Clear Signing for dApps – Get started"].

This design directly targets the current browser-wallet threat reality. WalletProbe reports that all 39 tested browser-based cryptocurrency wallet extensions could be abused to steal crypto assets [arXiv: "WalletProbe: A Testing Framework for Browser-based Cryptocurrency Wallet Extensions" (2025)]. WalletRadar reports 10 confirmed vulnerabilities from 8 wallet developers through automated analysis [arXiv: "WALLETRADAR: Towards Automating the Detection of Vulnerabilities in Browser-based Cryptocurrency Wallets" (2024)]. A 2025 study found that only 3 of 53 Ethereum wallets explicitly warned users under address-poisoning conditions [arXiv: "Ethereum Crypto Wallets under Address Poisoning: How Usable and Secure Are They?" (2025)]. The firewall therefore does three mandatory checks before any approval screen: detect whether the destination address matches recently observed poisoned patterns in the current interaction context, refuse blind raw-sign flows when an EIP-712 path exists, and downgrade unsupported advanced flows by capability detection instead of pretending the wallet can safely execute them [arXiv: "Ethereum Crypto Wallets under Address Poisoning: How Usable and Secure Are They?" (2025); EIP-5792 official specification; MetaMask Docs: "Sign data"].

The extension is essential because the browser platform already gives it the right control points: isolated content scripts, explicit injection permissions, and message-passing separation from the page [Chrome for Developers: "Content scripts"; Chrome for Developers: "Message passing"; Chrome for Developers: "The activeTab permission"]. Security is essential because recent evidence shows the attack surface is systemic, not exceptional [arXiv: "WalletProbe: A Testing Framework for Browser-based Cryptocurrency Wallet Extensions" (2025); arXiv: "WALLETRADAR: Towards Automating the Detection of Vulnerabilities in Browser-based Cryptocurrency Wallets" (2024)]. Smart contracts are essential because the safety upgrade comes from changing transaction semantics themselves: permit instead of approve, typed data instead of opaque payloads, bundled calls instead of fragmented approvals, and capability-advertised sponsorship instead of hidden execution assumptions [EIP-2612 official specification; EIP-712 official specification; EIP-5792 official specification; EIP-7677 official specification].

Interdependence proof:
- Remove browser extension: The system loses the isolated-world injector, activeTab-scoped page entry point, and message-passing choke point that lets it mediate page requests before they reach signing logic [Chrome for Developers: "Content scripts"; Chrome for Developers: "Message passing"; Chrome for Developers: "The activeTab permission"]. Then capability checks and typed-sign enforcement no longer sit at the browser interaction boundary.
- Remove security: The system becomes a convenience layer rather than a firewall. That fails the actual problem because current evidence shows widespread exploitable wallet-extension behavior, confirmed vulnerabilities across multiple developers, and weak warning coverage for address poisoning [arXiv: "WalletProbe: A Testing Framework for Browser-based Cryptocurrency Wallet Extensions" (2025); arXiv: "WALLETRADAR: Towards Automating the Detection of Vulnerabilities in Browser-based Cryptocurrency Wallets" (2024); arXiv: "Ethereum Crypto Wallets under Address Poisoning: How Usable and Secure Are They?" (2025)].
- Remove smart contract: The extension can no longer transform risky transaction structure. It loses permit-based approval reduction, EIP-712 human-readable signing, capability discovery, bundled calls, and paymaster-mediated execution [EIP-2612 official specification; EIP-712 official specification; EIP-5792 official specification; EIP-7677 official specification]. Then it can only warn about bad flows instead of replacing them with safer ones.

Failure mode:
If a wallet falsely advertises capabilities or falls back from typed, simulated, or clear-signable flows to opaque signing, the firewall collapses into a UI wrapper around the same unsafe approval path [EIP-5792 official specification; MetaMask Docs: "Sign data"; Ledger Developer Portal: "Clear Signing for dApps – Get started"].
If the extension requests broad host access instead of using activeTab-scoped injection and strict message mediation, it recreates the oversized browser attack surface that current wallet studies show is already widely exploitable [Chrome for Developers: "The activeTab permission"; arXiv: "WalletProbe: A Testing Framework for Browser-based Cryptocurrency Wallet Extensions" (2025)].

Open question:
How should a browser wallet formally verify that a dapp’s requested EIP-5792 or EIP-7677 execution path preserves the same user-visible intent as the human-readable EIP-712 or clear-signing prompt it shows before approval [EIP-5792 official specification; EIP-7677 official specification; EIP-712 official specification; Ledger Developer Portal: "Clear Signing for dApps – Get started"]?
