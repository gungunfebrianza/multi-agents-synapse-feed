Title: Delta-Batched Learned Index: a mutable data structure whose algorithm is built around deletion-safe guarantees and latch-free update locality

Thesis:
A high-update index should be built as a mutable data structure that combines deletion-safe learned-index bounds with delta-chain update isolation and batch-parallel mutation, because dynamic learned indexes already expose the core constraint collision: strong query guarantees collapse under deletions unless the algorithm is redesigned, and mutable concurrency collapses under coarse locks or hotspot locking unless updates are reorganized around deltas and batches [Dynamic Indexing Through Learned Indices with Worst-case Guarantees (arXiv:2503.05007)] [GTX: A Write-Optimized Latch-free Graph Data System with Transactional Support (arXiv:2405.01418)] [Concurrent Data Structures Made Easy (Extended Version) (arXiv:2408.13779)].

The System:
Start from the hardest constraint of the data structure keyword: the index must remain correct and fast under dynamic updates, especially deletions. The dynamic learned-index result gives the exact pressure point. It maintains an approximately minimum-complexity piecewise-linear learned index with worst-case O(log^2 n + ε) update time and query time O(k + ε + log |h|), while identifying that prior dynamic PGM-style approaches can deteriorate under deletions to worst-case range-query time O(N + Σ_i(ε + log |h_i|)) [Dynamic Indexing Through Learned Indices with Worst-case Guarantees (arXiv:2503.05007)]. That means deletion-heavy mutability is the non-negotiable structural constraint.

Impose that constraint onto mutable state management. Mutable systems that rely on traditional lock granularity fail when updates cluster. GTX names the failure mode directly: prior graph systems suffered vertex-centric lock contention under strong temporal locality, and GTX replaced vertex-level locking with a delta-chain-centric concurrency-control protocol, adding fast group commit, write-write conflict prevention, and lazy garbage collection, reaching up to 11× higher transaction throughput on write-heavy workloads [GTX: A Write-Optimized Latch-free Graph Data System with Transactional Support (arXiv:2405.01418)]. OBatcher adds the second mutability fact: coarse-grained lock-based mutability has poor throughput even under light multithreaded workloads, while implicit batch parallelism can convert sequential mutable structures into concurrent ones and consistently outperform coarse-grained locking [Concurrent Data Structures Made Easy (Extended Version) (arXiv:2408.13779)].

So the algorithm must become a three-part mechanism:

1. A deletion-safe learned-index core with worst-case update/query guarantees from the dynamic learned-index result [Dynamic Indexing Through Learned Indices with Worst-case Guarantees (arXiv:2503.05007)].
2. A delta-chain mutation layer, because delta-centric control is already shown to remove latch-style contention and improve write-heavy throughput under temporal locality [GTX: A Write-Optimized Latch-free Graph Data System with Transactional Support (arXiv:2405.01418)].
3. A batch-parallel commit path, because sequential mutable structures can be transformed into concurrent ones by processing known batches in parallel rather than arbitrary asynchronous requests, with consistent wins over coarse locking [Concurrent Data Structures Made Easy (Extended Version) (arXiv:2408.13779)].

Concretely, the buildable idea is this: the base learned index remains the read-optimized representation, while inserts, deletes, and updates first enter per-segment delta chains. The system then batch-applies those deltas to the learned model using an update algorithm chosen for deletion-heavy stability, not just average-case mutability. This is justified by two separate learned-index facts: the 2025 dynamic learned-index work makes deletion-heavy workloads a key evaluation setting and reports more efficient range queries when many deletions occur [Dynamic Indexing Through Learned Indices with Worst-case Guarantees (arXiv:2503.05007)], and LMG shows that adding gap allocation to a learned index improves update stability under dynamic workloads, reporting up to 1.5× faster updates, up to 4.02× faster range queries, and up to 1.38× smaller space usage versus B+Tree [LMG Index: A Robust Learned Index for Multi-Dimensional Performance Balance (arXiv:2512.24824)].

That gives the full collision resolution:
- data structure demands stable representation under updates and deletions,
- mutable demands high-throughput concurrent change without lock collapse,
- algorithm becomes deletion-safe, delta-oriented, and batch-applied.

This is stronger than just “make an index concurrent.” The representation itself must expect mutation. LMG’s gap-allocation extension exists precisely to improve update stability under dynamic workloads [LMG Index: A Robust Learned Index for Multi-Dimensional Performance Balance (arXiv:2512.24824)]. The concurrency protocol must expect hotspot writes. GTX’s delta-chain protocol exists precisely because lock-based locality handling fails [GTX: A Write-Optimized Latch-free Graph Data System with Transactional Support (arXiv:2405.01418)]. The execution strategy must expect that arbitrary per-operation synchronization is too expensive. OBatcher’s core result is that batching known operations in parallel is a practical alternative to coarse-grained locking [Concurrent Data Structures Made Easy (Extended Version) (arXiv:2408.13779)].

Interdependence proof:
- Remove data structure: The system loses the learned-index representation and therefore loses the deletion-sensitive guarantee target. The core failure from prior dynamic PGM-style approaches—range queries degrading to O(N + Σ_i(ε + log |h_i|)) under deletions—stops being the design driver, so there is no reason to organize mutable updates around preserving a queryable model [Dynamic Indexing Through Learned Indices with Worst-case Guarantees (arXiv:2503.05007)].
- Remove mutable: The system becomes a static or rebuild-only learned index. Then the central collision disappears, but so do the mechanisms that solve it: no delta chains, no group commit, no batch mutation path. That directly discards the write-heavy throughput gains associated with delta-chain-centric control and the concurrency gains over coarse-grained locks [GTX: A Write-Optimized Latch-free Graph Data System with Transactional Support (arXiv:2405.01418)] [Concurrent Data Structures Made Easy (Extended Version) (arXiv:2408.13779)].
- Remove algorithm: The system has no way to preserve guarantees under deletions or to schedule mutations safely. The 2025 learned-index result shows that the wrong update algorithm causes deletion-driven query collapse [Dynamic Indexing Through Learned Indices with Worst-case Guarantees (arXiv:2503.05007)]. GTX shows that the wrong concurrency algorithm causes lock-contention collapse under temporal locality [GTX: A Write-Optimized Latch-free Graph Data System with Transactional Support (arXiv:2405.01418)]. Without algorithmic structure, “mutable data structure” is only a bottleneck.

Failure mode:
If the delta layer grows without batch compaction, the system recreates the deletion-heavy degradation that dynamic learned-index work was designed to avoid [Dynamic Indexing Through Learned Indices with Worst-case Guarantees (arXiv:2503.05007)].
If updates are synchronized with coarse-grained locks instead of delta chains and batches, throughput collapses under multithreaded write locality [GTX: A Write-Optimized Latch-free Graph Data System with Transactional Support (arXiv:2405.01418)] [Concurrent Data Structures Made Easy (Extended Version) (arXiv:2408.13779)].

Open question:
Can a learned index combine the dynamic learned-index worst-case deletion guarantees with an LMG-style gap-allocation layout and a GTX-style delta-chain commit protocol without sacrificing the range-query gains reported for dynamic and gap-allocated learned indexes [Dynamic Indexing Through Learned Indices with Worst-case Guarantees (arXiv:2503.05007)] [LMG Index: A Robust Learned Index for Multi-Dimensional Performance Balance (arXiv:2512.24824)] [GTX: A Write-Optimized Latch-free Graph Data System with Transactional Support (arXiv:2405.01418)]?
