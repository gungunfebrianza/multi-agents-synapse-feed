Title: ActiveTab Contract Firewall: a browser extension that treats every smart-contract interaction as a pre-confirmation security verdict

Thesis:
The correct way to defend users from Web3 phishing is a browser extension that uses temporary tab access to capture wallet-bound transaction intent, independently decodes and simulates the smart-contract call before confirmation, and blocks or escalates risky approval and proxy patterns because the attack surface now lives in the seam between browser privileges, injected wallet providers, and contract semantics [Chrome for Developers: chrome.scripting API; Chrome for Developers: The "activeTab" permission; EIP-1193: Ethereum Provider JavaScript API; viem docs: decodeFunctionData; viem docs: simulateContract; EIP-20: Token Standard; OpenZeppelin Docs: ERC-1967 Proxy; Scam Sniffer Docs: Security Check].

The System:
Build a Manifest V3 browser extension that activates only on the user-invoked tab through `activeTab`, then injects inspection code with `chrome.scripting`, which requires both the `scripting` permission and matching host permissions [Chrome for Developers: chrome.scripting API; Chrome for Developers: The "activeTab" permission; Chrome for Developers: Declare permissions]. This matters because browser extensions are high-privilege software and can access broad client and network data, making extension minimization a security requirement, not a UX preference [CISA: Securing Web Browsers and Defending Against Malvertising for Federal Agencies; Chrome for Developers: Declare permissions].

The extension watches the page context where the wallet provider is exposed. EIP-1193 defines the provider as injected into an untrusted environment, says accounts should not be exposed by default, and requires explicit approval flows such as `eth_requestAccounts` or `wallet_requestPermissions`; it also recommends isolating wallet code, rate-limiting requests, and validating all provider-supplied data [EIP-1193: Ethereum Provider JavaScript API]. That makes provider-originated transaction requests inspectable security events.

When a transaction or signature request is initiated, the extension runs a local contract-intent pipeline before wallet confirmation, matching a pattern already used in production security tooling [Scam Sniffer Docs: Security Check]. First, it decodes calldata with `decodeFunctionData`; if the selector is unknown, it resolves signatures through public 4-byte databases now supported through 4byte.directory and Sourcify’s takeover of openchain.xyz’s 4byte APIs [viem docs: decodeFunctionData; 4byte.directory Signatures; Sourcify Docs blog]. Second, it classifies the target with `eth_getCode` and `eth_call`, and enriches with explorer metadata such as Etherscan’s `getsourcecode` fields including `Proxy` and `Implementation` [ethereum.org JSON-RPC API; Etherscan Docs: Get Contract Source Code]. Third, it resolves upgradeability using ERC-1967 implementation, admin, and beacon slots so the extension analyzes the effective logic contract, not just the proxy shell [OpenZeppelin Docs: ERC-1967 Proxy; OpenZeppelin Contracts 5.x Proxy API; OpenZeppelin Upgrades: Proxy Upgrade Pattern].

Then it simulates the exact call before signing. `simulateContract` supports eth_call-style validation, return data, and revert reasons without changing chain state, and Tenderly-backed simulation is already used by browser security products before wallet confirmation [viem docs: simulateContract; Scam Sniffer Docs: Security Check]. This simulation layer is mandatory because ScamSniffer documented transaction simulation spoofing as an active threat category and separately documented EIP-712 normalization bypasses that defeated some security alerts, proving that display-layer parsing alone is insufficient [Scam Sniffer Research archive: Transaction Simulation Spoofing: A New Threat in Web3; Scam Sniffer Research archive: Wallet Drainer Exploits Numerical Address Bypass Security Alerts].

The extension turns decoded and simulated results into contract-specific security rules. For ERC-20 approvals, it flags `approve` flows, reads allowance semantics from the standard, and escalates very large or max-style approvals because EIP-20 standardizes `approve`, `allowance`, and `transferFrom` and explicitly notes UI attack risk when changing an existing allowance, recommending clients set allowance to 0 before setting a new value [EIP-20: Token Standard]. For signatures, it does not trust normalized wallet displays and instead independently parses payloads and checks for known consequence patterns because normalized EIP-712 views have already been bypassed in drainer attacks [Scam Sniffer Research archive: Wallet Drainer Exploits Numerical Address Bypass Security Alerts].

The extension can also add browser-native containment around known phishing infrastructure. In MV3, arbitrary blocking logic in `webRequestBlocking` is no longer generally available, so network enforcement must be declarative through `declarativeNetRequest` [Chrome for Developers: chrome.webRequest API; Chrome for Developers: chrome.declarativeNetRequest API]. That means the system can ship fast deny or redirect rules for confirmed malicious domains while reserving programmable analysis for the transaction layer. This split is aligned with CISA guidance recommending browser and endpoint mitigations such as malicious-link protections, protective DNS, approved app stores, and remote browser isolation [CISA: Phishing Guidance – Stopping the Attack Cycle at Phase One (2025)].

A final safety module checks pasted destination addresses at the point of transaction composition. Microsoft warns malware or remote sessions can read from or place content into the clipboard, which maps directly to address-replacement attacks; the extension can detect pasted addresses and warn on mismatch with intended values [Microsoft Learn: Understanding security warnings when opening Remote Desktop (RDP) files; Microsoft Support: How malware can infect your PC]. In this design, browser extension, security logic, and smart-contract analysis are one system, not three features.

Interdependence proof:
- Remove browser extension: The system loses the privileged, in-browser interception point needed to inspect the active dapp tab, inject inspection code with `chrome.scripting`, and act before wallet confirmation; smart-contract decoding and simulation then happen too late or outside the user flow [Chrome for Developers: chrome.scripting API; Chrome for Developers: The "activeTab" permission; Scam Sniffer Docs: Security Check].
- Remove security: The system becomes a passive decoder. That fails against documented phishing losses, simulation spoofing, EIP-712 alert bypasses, malicious extensions, and clipboard replacement attacks because no verdicting or containment layer remains [Scam Sniffer Research archive: 2024 Web3 Phishing Attacks report; Scam Sniffer Research archive: Transaction Simulation Spoofing: A New Threat in Web3; Scam Sniffer Research archive: Wallet Drainer Exploits Numerical Address Bypass Security Alerts; CISA: Securing Web Browsers and Defending Against Malvertising for Federal Agencies; Microsoft Learn: Understanding security warnings when opening Remote Desktop (RDP) files].
- Remove smart contract: The system cannot decode calldata, simulate writes, detect ERC-20 allowance abuse, resolve proxies, or classify code at target addresses. It would be blind to the actual consequences of wallet actions [viem docs: decodeFunctionData; viem docs: simulateContract; EIP-20: Token Standard; OpenZeppelin Docs: ERC-1967 Proxy; ethereum.org JSON-RPC API].

Failure mode:
If the extension trusts wallet-rendered or normalized signature displays instead of independently decoding and simulating, documented drainer bypasses defeat it [Scam Sniffer Research archive: Wallet Drainer Exploits Numerical Address Bypass Security Alerts].
If it analyzes proxy shells without resolving ERC-1967 implementation addresses, upgradeable contracts hide the logic that actually executes [OpenZeppelin Docs: ERC-1967 Proxy; OpenZeppelin Contracts 5.x Proxy API].

Open question:
How should a browser extension prove that its own pre-confirmation simulation has not been spoofed when simulation spoofing is already a documented attack category and MV3 limits arbitrary in-path request logic [Scam Sniffer Research archive: Transaction Simulation Spoofing: A New Threat in Web3; Chrome for Developers: chrome.webRequest API; Chrome for Developers: chrome.declarativeNetRequest API]?
